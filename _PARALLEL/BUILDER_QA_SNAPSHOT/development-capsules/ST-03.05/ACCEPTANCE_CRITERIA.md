# Acceptance criteria

## AC-01 — Exact completed dependency

Given the ST-03.04 completion receipt and stored artifact set, when validation begins, then the receipt hash and canonical payload, active manifest, 21 artifact hashes, HarnessIR binding, and run reference all match exactly.

## AC-02 — Governed policy identity

Given the accepted constitutional-precedence contract, when it is loaded, then its path, SHA-256, Constitution V1.1 identity/hash, Builder PRD V1.2 identity/hash, authority order, conflict behavior, orders, lineage rule, and ownership boundaries are exact; altered bytes fail closed.

## AC-03 — Complete resolved artifact set

Given the active manifest, when cross-artifact validation runs, then exactly 8 human, 3 OpenSpec, and 10 machine views exist, every content hash is valid, and every declared source node resolves to the active HarnessIR.

## AC-04 — HarnessIR semantic agreement

Given every projected governed value, when compared with HarnessIR, then value, provenance, knowledge status, authority status, evidence, version, decision source, confidence, and disposition remain semantically identical; formatting does not create meaning.

## AC-05 — Constitutional precedence

Given a lower-authority generated view, when it conflicts with Constitution V1.1, Builder PRD V1.2, accepted policy, or HarnessIR, then the validation report is `FAIL` and no passing receipt or active run reference is committed.

## AC-06 — Rich lineage and applicability

Given the category-neutral synthetic proof, when lineage is checked, then all five rich Activative lineage keys remain separate and explicit `NOT_APPLICABLE` values remain distinguishable from missing, flattened, or invented values.

## AC-07 — Subordinate non-executable views

Given OpenSpec, graph, skill, evaluation, repair, dashboard, fixture, traceability, or ticket views, when authority is checked, then each remains non-executable and subordinate to HarnessIR and authorizes no later Story or external behavior.

## AC-08 — Syntax-valid conflict fails closed

Given syntactically valid Markdown or JSON with a missing reference, altered value, authority escalation, semantic compression, or downstream invention, when validated, then exact typed findings identify the artifact and IR nodes and readiness remains blocked.

## AC-09 — Authority and isolation

Given a validation command, when an agent, evaluator, external actor, unauthenticated human, network path, secret, arbitrary policy path, or unpinned policy is introduced, then validation fails without governed mutation or external side effect.

## AC-10 — Atomicity, replay, and idempotency

Given a valid report command, when committed or repeated, then the report, run event/reference, command record, and receipt commit atomically and repeat payloads return the same receipt; changed payload, stale version, or injected failure leaves no partial state.

## AC-11 — Invalidation and history

Given a validated artifact set, when the authoritative boundary is reopened, then HarnessIR, artifact set, and constitutional validation descendants become unusable while immutable history remains available and a new version is required.

## AC-12 — Evidence and bounded completion

Given completion, when the Story receipt is issued, then all required observations, findings, hashes, tests, rollback evidence, exact file scope, and 147-test predecessor regression are PASS, with no capability graph, Atomic Harness Definition, Development Capsule compiler, external runtime, or production claim.
