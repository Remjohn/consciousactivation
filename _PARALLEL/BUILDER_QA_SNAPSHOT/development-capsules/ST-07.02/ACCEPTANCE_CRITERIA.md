# Acceptance Criteria

## AC-01 — Compile the exact generic Atomic Content Harness definition

Given the exact active governed synthetic lineage and authorized compiler command, when compilation runs, then one immutable `AtomicHarnessDefinition` for `atomic_content_harness` is committed with every capsule-required section and receipt.

## AC-02 — Preserve target-specific source and IR identity

Given the repository-owned synthetic profile, Source Lock, and active Harness IR, when the definition is inspected, then their exact identities, versions, hashes, canonical bytes references, provenance, and authority are preserved without Format 02 or external sources.

## AC-03 — Preserve pre-ratified Genesis and constitutional authority

Given the frozen boundary, ratification, Draft Harness Model, and active constitutional-precedence evidence, when compilation resolves authority, then only those exact active decisions govern and no local rule weakens or replaces them.

## AC-04 — Compile a complete non-executing Builder plan

Given the active capability graph, responsibility modules, Phase Graph, internal handoff, and Minimum Complete Context, when compiled, then the definition contains complete typed references and a governed execution-plan declaration without executing a phase or workflow.

## AC-05 — Prove the skill boundary

Given the active zero-skill snapshot and ST-05.02 necessity decision, when compiled, then the definition records zero external/missing/adaptation/experiment/JIT requirements and rejects absent, altered, or contradictory skill evidence.

## AC-06 — Preserve semantic distinctions and synthetic scope

Given a category-neutral proof, when the definition is serialized, then category adapter and category-native semantics are explicitly `NOT_APPLICABLE_SYNTHETIC_PROOF`, `synthetic_not_certifiable` is true, and nonproduction/noncertified states cannot be omitted or upgraded.

## AC-07 — Compile target/output and acceptance-test contracts

Given the governed deterministic text-normalization task, when compiled, then its atomic input/output boundary, required context, capability declarations, phase plan, and executable acceptance-test declarations are present without implementing or executing the task.

## AC-08 — Fail closed on invalid inputs or authority

Given missing, stale, altered, superseded, invalidated, category-bound, externally owned, or unauthorized input, when compilation is attempted, then a typed rejection occurs and no partial definition, receipt, event, observation, command record, or run reference is committed.

## AC-09 — Be deterministic and immutable

Given identical governed inputs in fresh contexts, when compiled, then definition bytes, hashes, identity, ordering, and receipt are identical; a changed governed input requires a new immutable identity and existing history is never mutated.

## AC-10 — Preserve replay and idempotency

Given a completed command, when repeated or restored through checkpoint replay, then the original receipt and run-state identity are returned without duplicate state; a conflicting payload fails closed.

## AC-11 — Propagate invalidation non-destructively

Given an upstream artifact is reopened or invalidated, when cascade rules run, then the active definition becomes unusable through a new descendant invalidation while historical bytes and receipts remain reproducible.

## AC-12 — Respect architecture and product boundaries

Given the final change set, when architecture and scope tests run, then only allowlisted files changed; no external dependency, schema, database, network, runtime, Format 02, category adapter, VAE, Delegation, GPU, conversational, Control Tower, Development Capsule generation, production certification, or later-Story behavior exists.
