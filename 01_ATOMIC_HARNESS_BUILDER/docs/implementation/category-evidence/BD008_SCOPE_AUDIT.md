## Repository search and candidate screening

The complete repository-local search covered `sources/`, `governance/`,
`contracts/`, `skill-packages/`, `evaluation/`, `tests/fixtures/`,
`development-capsules/`, `prd/` and `docs/`. Only the following plausible inputs
were found:

1. Constitution V1.1 section 16 contains named Julie and James end-to-end examples.
   The Julie example has a source premise, four Identity DNA keywords, one audience
   premise, one edge, activation directions/roles/stance and an Activative Call.
   It does not provide an immutable real-profile identity, profile-owner usage
   permission, an independently hash-bound Identity DNA snapshot, complete audience
   context, Resonance Map, Matrix of Edging, desired-reaction authority,
   micro-commitment authority, wrong-reading locks or field-level lineage. James is
   explicitly tentative and is still less complete. The Constitution is authority
   for semantics; its examples are not silently promoted into operator-authorized
   real profiles.
2. `evaluation/.../BASE_CASE.json` is expressly a public development fixture and its
   corpus is ineligible for production certification. It may verify code behavior,
   but it is not real-profile evidence.
3. `tests/fixtures/productization/manifests/activative_expression.json` is a test
   fixture with obvious repeated placeholder hashes. It is not a governed source or
   real profile.
4. `governance/SOURCE_REGISTER.json` records the historical Activative doctrine
   archive as `SRC-003`, but its `/mnt/data/...zip` path is inaccessible from the
   current workspace. The only accessible extracted SRC-013 artifact is the
   Constitution. Archive metadata alone cannot establish profile authority, usage
   rights or the required field lineage.

No candidate satisfied the eligibility gate. No real-profile compiler run was
performed, because doing so would require inventing or silently approving human
truth and provenance.

## Eligibility gate

One eligible input must include all of the following before any
`development_validated_real_profile` result can be issued:

- a non-synthetic immutable profile identifier and version;
- the profile owner or authorized steward;
- explicit permission for Builder development validation;
- original source material and exact source spans;
- immutable Identity DNA, Audience Context, Resonance Map and Matrix of Edging
  references;
- governed Edge Pressure, format goal, desired human/viewer role, desired reaction,
  micro-commitment, Activative Call constraints and non-empty wrong-reading locks;
- field-by-field source and authority lineage;
- explicit applicability decisions for Reaction Receipt, Expression Moment, Visual
  Semantic Pack, Visual Narrative Program, Feature Contracts, T/V route and
  Composition Intent;
- an explicit statement that absent human reaction or expression artifacts are not
  being manufactured;
- hash-valid source bytes and a revocation/amendment path.

The precise input package and manifest are defined in
`BD008_REQUIRED_OPERATOR_INPUT.yaml` and
`BD008_REAL_PROFILE_INPUT_MANIFEST_TEMPLATE.yaml`.
