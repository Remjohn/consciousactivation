# Acceptance Criteria

## AC-01 — Exact active dependency

Given the independently valid ST-05.01 snapshot and PASS receipt, when deterministic Builder code runs the necessity test, then exactly one immutable decision and receipt are committed with exact run, context, registry, policy, and five-capability lineage.

## AC-02 — Formal five-capability gap test

Given the five capsule-defined capability requirements, when they are evaluated, then each is assessed exactly once, has no observed target failure, and is proven satisfied by its explicit deterministic Builder-code owner.

## AC-03 — Governed alternatives precede new-skill work

Given the necessity policy, when alternatives are assessed, then deterministic code, schema/validator, tool, inline instruction, external reference, human decision, canonical reuse, local adaptation, and adapter composition are recorded in governed order before new canonical skill.

## AC-04 — No-skill outcome

Given the exact synthetic evidence, when the aggregate decision is compiled, then its outcome is `NO_NEW_SKILL_REQUIRED`, external and dynamic skill counts remain zero, and no skill, adaptation, recipe, binding, package, or JIT capsule is created.

## AC-05 — Skill Design Brief disposition

Given no demonstrated gap, when the decision is emitted, then the typed brief disposition is `NOT_APPLICABLE_NO_GAP` and no fabricated brief exists; any future gap or material-adaptation decision without a complete typed brief and human authority fails closed.

## AC-06 — Anti-goal enforcement

Given a proposed skill for behavior already owned by code, schema, workflow, human authority, reference, or an adequate reusable capability, when validation runs, then the proposal is rejected with the selected alternative owner and no partial state.

## AC-07 — Authority and version boundaries

Given an unauthorized actor, altered snapshot, stale context, same-version mutation, or request to design/register/discover/package/evaluate/execute a skill, when submitted, then it fails closed and requires new governed upstream authority where applicable.

## AC-08 — Determinism and idempotency

Given identical governed inputs in fresh repositories, when the test runs, then decision bytes, identities, receipt bytes, observations, and hashes are identical; an identical repeat returns the original receipt and a conflicting payload fails closed.

## AC-09 — Replay, resume, and invalidation

Given committed necessity events and receipts, when the run is replayed or resumed, then state reconstructs exactly; upstream invalidation makes the decision inactive while historical bytes remain reproducible.

## AC-10 — Atomic failure and rollback

Given injected failure at the authorized commit boundary, when compilation fails, then no decision, receipt, event, command record, run reference, or descendant state is partially committed and retry succeeds after the fault is removed.

## AC-11 — External and later-story exclusions

Given the synthetic mode, when architecture tests inspect the implementation, then no Format 02, VAE, Delegation runtime, provider, GPU, conversational, real skill registry, skill package, JIT capsule, workflow execution, Control Tower, or production behavior exists.

## AC-12 — Observable completion

Given all acceptance and regression tests pass, when completion evidence is issued, then success, replay, rejection, and invalidation observations include run, Story, decision, snapshot, capability coverage, alternative assessments, authority, version, provenance, outcome, and failure context in a separate hash-pinned PASS receipt.
