# Acceptance Criteria

## AC-01 — governed generation

Given the exact four predecessor PASS receipts and active validated synthetic Harness Definition, when deterministic Builder code generates the Development Capsule, then exactly one immutable versioned capsule and receipt are committed.

## AC-02 — complete requirement trace

Given D029 and FR-151 through FR-155, when the capsule is validated, then every owned obligation appears exactly once with acceptance and implementation evidence.

## AC-03 — authority and lineage

Given the canonical Constitution, Builder V1.2 authority and active upstream artifacts, when the capsule is generated, then their identities and hashes are preserved without local constitutional redefinition.

## AC-04 — justified scaffolding

Given the synthetic Harness scope, when scaffolding is included, then every item has an implementation reason and no unrelated framework, service, database, runtime or external-product scaffold appears.

## AC-05 — contracts and examples

Given the AtomicHarnessDefinition, when the capsule is generated, then its typed contracts, compatibility state, examples and synthetic fixtures are hash-pinned and portable.

## AC-06 — executable tests

Given the accepted requirements, when the capsule is validated, then deterministic acceptance, negative, observability, rollback and architecture tests are explicitly referenced.

## AC-07 — implementation completeness

Given a fresh implementation agent, when only the generated capsule is supplied, then exact scope, dependencies, tests, observability, rollback, invalidation and completion receipt requirements are available without invented product decisions.

## AC-08 — fail-closed generation

Given incomplete traceability, unresolved contract versions, unjustified scaffolding, missing fixtures/tests, mutable inputs, external behavior, or production claims, when generation is attempted, then typed rejection occurs with zero partial state.

## AC-09 — authority

Given an actor without deterministic Builder-code authority, when generation, replacement or invalidation is attempted, then the action fails closed.

## AC-10 — determinism and portability

Given identical governed inputs in fresh contexts, when capsules are generated, then canonical bytes, hashes, ordering and receipt identities are identical and contain no absolute local path.

## AC-11 — replay and invalidation

Given a repeated command or invalidated upstream artifact, when the service processes it, then the original receipt is returned for an identical payload and affected active capsules are invalidated without destroying history.

## AC-12 — bounded classification

Given the synthetic proof, when the capsule is inspected, then it remains repository-owned, non-production, non-certified, `synthetic_not_certifiable`, and requires zero external skills and runtimes.
