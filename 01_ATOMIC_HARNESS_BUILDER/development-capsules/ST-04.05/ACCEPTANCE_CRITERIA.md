# Acceptance criteria

## AC-01 — Compile minimum-complete phase contexts

Given the exact active accepted ST-04.04 internal handoff and hash-pinned synthetic context input, when authorized Builder code compiles ST-04.05, then both governed phases receive one immutable minimum-complete context manifest and one attributable completion receipt.

## AC-02 — Register every reference explicitly

Given every required, optional, excluded, and pointer resource, when compilation succeeds, then each resource has a stable ID, version, runtime-resolved or literal hash, owner, authority, content role, loading mode, and influence boundary; ghost or prose-only references fail closed.

## AC-03 — Apply loading and influence boundaries

Given phase-local loading rules, when manifests are compiled, then only allowlisted resources are included in their permitted phases, prohibited and must-not-influence resources remain explicitly excluded, and no resource acquires semantic or constitutional authority through loading.

## AC-04 — Compile progressive-disclosure pointers

Given branch-specific or heavy material, when it is not universally required, then the manifest records an immutable typed pointer and rationale rather than loading its payload; a missing, altered, or unregistered pointer target blocks compilation.

## AC-05 — Keep SPR governed and phase-local

Given the synthetic deterministic proof has no creative model phase, when context is compiled, then SPR remains explicitly `NOT_APPLICABLE` and excluded, cannot rewrite evidence or authority, and no general claim about production SPR requirements is made.

## AC-06 — Rank by functional necessity

Given required and optional context classes, when selection is compiled, then phase responsibility, output contract, ratified decisions, bindings, canonical procedure, adaptation, direct evidence, constraints, conditional references, examples, and enrichment retain the governed priority order unless a new authorized version records a justified override.

## AC-07 — Enforce complete explicit budgets

Given each phase policy, when compilation occurs, then hard and soft token, latency, and cost budgets, required and optional classes, compression permissions, retrieval policy, and overflow behavior are visible and validated without invoking a tokenizer, model, provider, or runtime.

## AC-08 — Block required overflow without truncation

Given required resources exceed a hard budget, when compilation is attempted, then the Builder commits no context graph or manifest and emits the exact overflowing resource identities plus governed remediation choices; no required resource is cut, omitted, or silently summarized.

## AC-09 — Emit complete reproducible manifests

Given a successful compile, when a reviewer inspects each phase manifest, then included, excluded, summarized, retrieved, and compressed resources, exact hashes, governed token contributions, priority and decision rationale are complete and deterministically reproducible.

## AC-10 — Preserve lineage, replay, and invalidation

Given identical inputs and command payload, when compilation repeats or resumes, then it returns identical graph, manifest, and receipt bytes without duplicate state; changed references or upstream invalidation create new immutable identity or invalidate only affected descendants while history remains reproducible.

## AC-11 — Keep runtime and external boundaries closed

Given the Builder Core synthetic mode, when ST-04.05 runs, then no conversation history, model, remote retrieval, skill execution, JIT capsule, phase/workflow execution, Format 02, VAE, Delegation runtime, GPU, provider, evaluator, Control Tower, production, or publication behavior is invoked or claimed.

## AC-12 — Produce bounded completion evidence

Given all tests and regressions pass, when completion is evaluated, then deterministic observations, overflow and rollback evidence, exact changed-file hashes, scope validation, and a separate canonical Story Completion Receipt are present; ST-05.01 remains unauthorized.
