# Product Requirements Document - CMF STUDIO

**Author:** Emilio
**Date:** 2026-06-19

---

## Executive Summary

CMF STUDIO is an interview-first expression operating system for transforming researched, authentic human expression into coherent, reusable, evaluated, approved, and publishable media assets. It is not an editor-first clipping tool and not a generic AI content generator. Its core premise is that the quality ceiling of a produced asset is set before editing begins: by the depth, specificity, emotional charge, and routeability of the human expression captured in the source session.

The system begins upstream with research, audience reality, Matrix of Edging pressure mapping, Narrative State Induction, First-Line Anchors, Depth Anchors, and Interview Asset Contracts. These components help the Operator guide experts away from safe, abstract "centroid answers" and into production-ready expression moments that can be deterministically routed into valid content archetypes and asset derivatives.

CMF STUDIO is also a greenfield continuity project. The legacy repository remains read-only, but its high-value intelligence must be migrated into the new Python-first Harness: cognitive primitives, SDA/SFL registries, archetype prompts, creative subsystem configs, Voice DNA verification, audio composition behavior, CBAR gates, ERA3 spec protocols, TTT tonal methodology, ComfyUI templates, fixtures, and evals. No legacy runtime code may be imported directly into production.

Downstream, CMF STUDIO operates as a Python-first agentic production harness. Pydantic v2 owns semantic contracts, DSPy owns structured reasoning and calibrated evaluations, Pi Coding Agent orchestration operates through approved tools and typed commands, and TypeScript remains limited to presentation and rendering surfaces such as the Next.js PWA, Telegram Mini App, Remotion, and Motion Canvas.

The product must be built as the full documented system, following dependency-ordered build gates and production release gates. It should not be scoped as a partial product, because the system's value depends on interlocking contracts across legacy migration, Brand Genesis, expression capture, archetype routing, rendering, evaluation, publishing, and memory.

### What Makes This Special

CMF STUDIO's differentiator is a complete expression-to-asset factory, with Narrative State Induction as the upstream ignition point. The system does not merely find good clips after a recording. It prepares the conditions for better source material, captures that expression with lineage, routes it through schema-defined archetypes, renders it using a locked brand universe, evaluates it through calibrated critics, and preserves its memory for future production.

The moat is created by the collision of several doctrines that normally live in separate tools:

- Narrative State Induction creates stronger source expression before recording.
- Matrix of Edging turns research pressure into usable interview and routing tension.
- Brand Genesis locks a reusable 64-state acting library, paper-cut avatar rig, micro-semiotic anchor library, and Brand Context Version.
- Archetype routing prevents generic formats by mapping expression into validated structures such as Core Educator, Cinematic Story, Myth Debunk, Persuasive Micro-Claim, Thought Whisperer Extract, Data Story Post, carousels, meme visuals, poll visuals, reaction formats, and paper-cut render routes.
- Legacy intelligence is preserved through explicit migration into Pydantic registry contracts, DSPy programs, worker assets, fixtures, and evaluation gates.
- Complete Editing Sessions preserve provenance from source expression to rendered asset.
- SemanticCritic, ImageCritic, VoiceContinuityCritic, anti-draft calibration, CBAR, and TTT-style tonal evaluation enforce fidelity, identity consistency, emotional accuracy, negative space compliance, motion restraint, and routeability.
- The Operator remains the final arbiter through the PWA Control Tower and Telegram Operator Cockpit.
- Publishing and memory are handled through explicit approval events, Publer integration, and long-term brand/interviewer memory.

This makes CMF STUDIO fundamentally different from post-production tools. It controls the full path from authentic human activation to approved media output without losing identity, truth, visual consistency, legacy intelligence, or auditability.

## Success Criteria

### User Success

CMF STUDIO succeeds for the Operator when one real brand cycle can be completed end-to-end without developer intervention, manual database edits, direct provider tinkering, hidden prompt surgery, or direct legacy runtime imports.

A successful Operator workflow includes:

- onboarding a brand;
- collecting client consent and source materials;
- generating and approving Brand Genesis assets;
- preparing Interview Intelligence from research, audience reality, Matrix of Edging, and migrated legacy primitives;
- conducting or supporting a Complete Expression Session;
- extracting routeable expression moments;
- compiling Complete Editing Sessions;
- rendering evaluated assets across valid CMF routes;
- reviewing and revising through the PWA Control Tower or Telegram Operator Cockpit;
- approving publishing intent;
- scheduling approved output through Publer;
- updating brand, interviewer, route, and performance memory.

The Operator should feel that CMF STUDIO has absorbed the old manual burden of hunting, diagnosing, composing, and reinventing the brand, while preserving human judgment, interview presence, taste, and final approval authority.

User success also requires validated use of migrated legacy intelligence. The 244 cognitive primitives, SDA/SFL registries, 96 archetype prompts, 34 creative subsystems, Voice DNA checks, CMF engine references, CBAR gates, ERA3 spec protocols, and TTT tonal methodology must influence the actual Operator workflow as typed, evaluated, auditable assets.

### Business Success

Business success is measured by migration-backed production capacity: CMF STUDIO increases the number of approved, publishable, client-trust-preserving asset packs one Operator can complete per month without lowering quality.

Success is not raw content volume or generic SaaS user growth. Output only counts if it preserves:

- source truth;
- brand fidelity;
- routeability;
- voice authenticity;
- evaluation receipts;
- human approval;
- lineage from source expression to publication;
- no direct legacy runtime dependency.

At the first readiness checkpoint, success means Build Gate 0 and early contract readiness are materially proven: the legacy repository inventory is classified, high-value migration targets are defined, foundational Pydantic contracts exist, initial DSPy/eval fixtures are specified, and the system can simulate or execute slices of the core brand-cycle path without violating architecture.

At the operational maturity checkpoint, success means multiple real brand cycles have been completed through the system, with publishable asset packs produced across valid archetype routes, revision loops traceable, Publer scheduling approval-gated, memory improving future sessions, and Operator effort shifted from manual editing toward creative direction and approval.

### Technical Success

Technical success is architectural integrity under production pressure.

The system succeeds technically when:

- Pydantic v2 is the canonical contract authority for commands, events, registry entries, workflow inputs, provider jobs, receipts, and evaluation outputs.
- DSPy owns structured reasoning, extraction, routing, calibration, and evaluation programs where applicable.
- Pi orchestrates through approved tools and typed commands, never by directly mutating production state.
- Durable workflows own retries, waits, provider polling, resumability, cancellation, compensation, and human approval waits.
- TypeScript is restricted to generated contract consumers and presentation/rendering surfaces.
- Every state-changing action passes through the Command Bus.
- Every render or publishable output has a lineage receipt from source truth to publication.
- Public publishing requires explicit human approval.
- Provider failure, worker failure, and workflow interruption can recover without losing completed work.
- Frontend and renderer types regenerate from Pydantic without semantic drift.
- Model providers can be replaced without rewriting domain rules.

### Pi Harness Orchestration Rules

Pi orchestrates the pipeline; it does not replace the pipeline. For each stage, Pi must read the active production object, identify the next valid state transition, select the correct role contract, JIT skill, DSPy program, provider adapter, deterministic service, or durable workflow, write a pre-execution `ValidationContract`, and propose a typed `AgentCommand`.

After execution, the stage must produce an honest receipt: success, failure, blocked action, partial output, human handoff, or quarantine. Pi may continue only when the receipt authorizes the next transition. Pi cannot directly mutate production tables, bypass consent, approve its own output, publish publicly, change locked Brand Context Versions, or treat Neo4j as canonical business state.

Every handoff between agents must carry an `AgentHandoffPacket` with the active object, source evidence, upstream receipts, required downstream object, allowed actions, blocked actions, and proof obligations. Every JIT skill or migrated compiler invocation must create a `SkillInvocationRecord` naming the source context, registry snapshot, compiler fingerprint, contrastive prompt layer, critic result, and synthesis result.

Agent autonomy is therefore bounded autonomy. Agents may choose how to complete a stage when multiple approved routes exist, but they may not reorder the production chain, skip validation, invent unsupported formats, publish without human approval, or collapse legacy orchestration modules into generic prompt labels.

## User Journeys

### Journey 1: Claude - The Expression-Latent Expert Becomes Legible

Claude is a thoughtful expert with a deep lived story, but when he tries to explain his work on camera, he drifts into history, theory, and careful abstractions. He has real material: identity between Burundi and Belgium, inherited silence, reconciliation, language, labels, exile, and memory. What he lacks is not insight. He lacks a conversational container that helps the strongest expression arrive cleanly.

Before the session, CMF STUDIO compiles his Guest Dossier, Audience Reality Brief, Matrix of Edging, interviewer resonance context, and Interview Asset Contracts. Claude does not begin inside a software interface. He begins with a human invitation and a clear recording protocol. The Operator confirms likeness consent, file-safety expectations, phone master recording, framing, lighting, audio, and upload path.

The breakthrough comes during the Complete Expression Session. Instead of asking generic biographical questions, the Operator uses First-Line Anchors and Depth Anchors to move Claude away from centroid answers and into production-ready expression. One answer becomes a Cinematic Story. Another becomes a Myth Debunk. Another becomes an Audience Mirror. Claude remains in control of truth, but the system preserves the moments with timestamped anchor hits, archetype candidates, and evaluation receipts.

Claude receives a Guest Asset Pack: 4 videos, 2 carousels, 2 meme visuals, 2 poll visuals, and reaction seeds. He sees that one meaningful conversation can become a week of dignified, specific content without flattening his story.

### Journey 2: Mira - The Operator Runs a Complete Expression Session

Mira is the internal Operator. She is not trying to become a faster video editor. Her job is to create the conditions for better source material and then direct the factory with taste. Before the session, she reviews the generated interview deck as a set of Interview Asset Contracts, not a loose question list. Each contract shows the target archetype, expression state, edge product, First-Line Anchors, Depth Anchor, clip rules, followups, asset routes, and evaluation targets.

During the call, Mira sees live guidance in the PWA Control Tower. Telegram stays available for operational notifications, but it is not forced onto the guest. Mira checks the recording quality gate: phone master recording, Meet backup, framing, audio, light, stability, file safety, and upload readiness. If Claude answers too historically, the system suggests a repair followup. If he lands somewhere stronger than predicted, the system records that landing rather than forcing a scripted ending.

The critical value moment is that Mira can steer a high-stakes interview without manually holding the whole production graph in her head. CMF STUDIO tracks anchor success, depth success, guest comfort, narrative density, clipability, and route coverage. After the session, she sees which moments can become Complete Editing Sessions and which need no production.

### Journey 3: Imani - The Production Steward Compiles Assets From Expression

Imani takes over after the session. She opens the extracted expression moments and sees each one connected to source timestamps, transcript evidence, archetype candidates, CMF render routes, Brand Context Version, and evaluation requirements. She is not writing prompts from scratch. She is compiling assets from validated contracts.

For Claude's session, Imani selects one Cinematic Story Commentary, one Educational Explainer, one Challenger or Frame Breaker clip, and one Reaction or Recognition clip. She also prepares two carousels, two meme visuals, two poll visuals, and stores reaction seeds for later. The system uses migrated legacy intelligence where appropriate: archetype prompt families become DSPy templates, cognitive primitives inform premise and extraction, SDA/SFL registries support sonic and compression standards, and CMF engine references guide captioning, EDL structure, timeline assembly, audio ducking, and render adapter behavior.

The breakthrough is consistency. Imani can move from source expression to renderable Complete Editing Sessions without losing lineage. If a render fails, the workflow retries or compensates. If a route is weak, the evaluation receipt explains why. If a voice segment sounds generic, anti-draft calibration and Voice DNA checks flag it before the asset reaches approval.

### Journey 4: Rafael - The Reviewer Protects Truth, Dignity, and Publishing Safety

Rafael is responsible for approval. He receives completed assets with source lineage, Brand Context Version, archetype route, evaluation receipt, transcript evidence, and publish intent. His job is not to inspect every technical detail manually. His job is to decide whether the asset is true, dignified, on-brand, platform-fit, and safe to schedule.

One asset looks visually strong but slightly overstates Claude's relationship to reconciliation work. Rafael rejects it with a note: preserve the emotional force, but soften the claim. The system creates a revision command rather than silently editing state. Another asset has strong timing but weak tonal continuity between segments. Rafael triggers a Voice-DNA Boost Insert or requests an editorial cutaway. The system records the decision and updates route memory only from evidence.

The value moment is approval safety. Nothing reaches Publer because an agent inferred intent. Publishing requires explicit human approval. Rafael can approve, reject, revise, or escalate from the PWA, with Telegram used for lightweight review when appropriate.

## Innovation & Novel Patterns

### Detected Innovation Areas

CMF STUDIO's innovation is not generic AI video generation. Its core novelty is a contract-governed expression factory that begins before recording and continues through extraction, routing, rendering, approval, publishing, and learning memory.

The first innovation area is **dual-layer extraction**. CMF STUDIO extracts first from the guest, then from the transcript. The first layer is Narrative State Induction: research, interviewer resonance, audience reality, Matrix of Edging, Interview Asset Contracts, First-Line Anchors, Depth Anchors, and live Operator guidance help the guest access clearer, more specific, more emotionally charged expression while preserving truth. The second layer is artifact extraction: after capture, the system extracts Expression Moments from transcript, timestamps, audio/video evidence, anchor hits, emotional shifts, archetype candidates, and evaluation receipts.

The second innovation area is **compiled narrative extraction intelligence**. The legacy JIT Skill Compiler and skill-module ecosystem should be migrated as a distinct moat. CMF does not rely on generic prompt improvisation. It compiles context-specific narrative capability from archetype templates, dependency registries, adapter registries, module libraries, Voice DNA, negative space, psychological routing, CRAL findings, anti-draft constraints, and fingerprint archives. These compiled lenses guide induction, extraction, drafting, evaluation, and learning.

The third innovation area is **Interview Asset Contracts as production infrastructure**. In CMF STUDIO, an interview deck is not a list of questions. Every content-intended question carries target archetype, expression state, anchors, depth path, asset derivatives, CMF routes, followups, clip rules, and evaluation logic. This turns the interview into a structured expression capture environment without scripting the guest.

The fourth innovation area is **governed composition and scene reproducibility**. Ideogram 4 `CompositionJob` contracts, legacy beat-fingerprint lineage, render manifests, prompt hashes, provider metadata, evaluation receipts, and approval state make visual production auditable. Ideogram is treated as a Composition Director, not a final identity renderer.

The fifth innovation area is **human-approved AI media operations**. CMF integrates Pydantic contracts, DSPy reasoning/evals, Pi orchestration, durable workflows, provider adapters, Telegram/PWA approvals, Publer scheduling, Neo4j projections, and receipt chains into one production system where humans retain truth, taste, and publication authority.

### Market Context & Competitive Landscape

Current AI media tools largely cluster around editor-first, repurposing-first, or generation-first workflows. Descript focuses on AI-assisted editing, transcript-based editing, clips, captions, AI speech, and media generation. OpusClip focuses on turning long videos into short clips with captions, reframing, B-roll, scheduling, and automation. Riverside combines recording, editing, repurposing, social clips, transcription, and publishing. Runway pushes generative media creation.

CMF STUDIO challenges the shared assumption behind many of these tools: that the source material already contains enough expression and the product's job is to find, polish, or generate the best output from it. CMF's premise is different. It assumes the strongest asset is often not present until the guest is properly activated. The system therefore competes less as "another video editor" and more as an expression-to-asset operating system.

The legacy skill/compiler layer deepens the separation. Generative tools often focus on scripts, prompts, shots, clips, or B-roll. CMF uses compiled narrative and extraction modules to influence what gets asked, how the guest is guided, what kind of answer becomes extractable, what moment survives evaluation, and how it routes into an asset package.

### Validation Approach

CMF STUDIO must validate the innovation through evidence, not novelty language.

First, validate **live expression improvement**. Compare normal expert interviews against CMF-guided Complete Expression Sessions. Measure truth density, emotional specificity, non-centroid phrasing, quoteability, archetype routeability, guest comfort, anchor success, depth success, and landing strength.

Second, validate **artifact extraction quality**. From each Complete Expression Session, the system should identify Expression Moments with transcript evidence, timestamps, anchor hits, source-truth scores, archetype candidates, and asset derivative potential. The measure is not raw clip count; it is approved, truthful, routeable asset yield.

Third, validate **JIT compiler migration value**. Migrated skill modules, adapter logic, anti-draft calibration, Voice DNA, negative-space constraints, CRAL routing, and fingerprint archive mechanics must improve extraction and drafting outcomes compared to generic DSPy prompts. Evaluation should compare generic extraction against compiled extraction lenses on the same transcript.

Fourth, validate **Guest Asset Pack fulfillment**. A successful session should support the documented package when source material allows it: 4 videos, 2 carousels, 2 meme visuals, 2 poll visuals, and 2-3 reaction seeds. Each deliverable must trace to source expression, archetype route, CMF route, evaluation receipt, and approval state.

Fifth, validate **governed reproducibility**. Operators and reviewers must be able to inspect why an asset exists: source moment, Interview Asset Contract, Brand Context Version, SceneSpec, Ideogram `CompositionJob`, render contract, evaluation receipts, revision history, and human approval.

Sixth, validate **production throughput**. One Operator should complete more approved, on-brand, truthful asset packs per month with less manual timeline work and fewer untraceable decisions.

### Risk Mitigation

The main innovation risk is over-automation of the human field. Narrative State Induction must never become coercive scripting. The guest controls truth, and the system must preserve authentic speech even when a predicted route fails. If the guest lands somewhere better than the plan, the system records the better landing rather than forcing the preplanned one.

The second risk is mistaking compiled skill intelligence for magic. JIT skill modules must be migrated as typed contracts, DSPy programs, fixtures, evals, and receipts. If a legacy module cannot be validated, it remains reference or future-review material. Direct legacy runtime imports are blocked.

The third risk is extraction overfitting. The system could become too eager to force every answer into an archetype. Mitigation: expression moments require source truth, archetype fit, guest dignity, and evaluation receipts. Weak routes should be rejected or marked for memory, not turned into deliverables.

The fourth risk is anti-draft overload. Negative-space and contrastive prompting can paralyze generation if too broad. The PRD should require pruning, adapter receipts, critic enforcement, and performance-based calibration, not endless forbidden lists.

The fifth risk is provider spectacle replacing brand truth. Ideogram 4, Flux 2 Klein 9b, GPT Image 2, Qwen-Image-Layered, SAM3, Remotion, and Motion Canvas must remain behind provider and renderer contracts. Visual generation cannot override Brand Context Version, source truth, text-area requirements, human approval, or composition receipts.

The fallback strategy is graceful reduction of autonomy. If compiled extraction or automated routing underperforms, CMF still preserves the interview-first workflow, contracts, transcripts, receipts, PWA review, and manual Operator routing. The system can operate with more human steering while evaluators and compiled modules are calibrated.

This PRD does not define later feature buckets. Documented CMF capabilities are not deferred simply because they are complex.

Later work may increase capacity, coverage, and polish only after the documented system is functioning, but it must not be framed as the place where the real product appears. Future additions require separate documentation and approval.

### Explicit Out of Scope

The following are excluded from this PRD unless separately documented and approved:

- newsletters as a CMF deliverable;
- generic self-serve content SaaS;
- unsupported archetype formats;

- RunningHub dependency;
- public marketplace;
- generic CRM/Zapier-style automation;
- autonomous public replies;
- broad external compliance/certification positioning;

### Risk Mitigation Strategy

Technical risk is handled through release gates, not feature removal. The system must enforce migration ledger discipline, Pydantic source-of-truth contracts, DSPy evals, Command Bus, durable workflows, provider capability registry, receipt chains, human approval gates, generated TypeScript consumers, direct-import blockers, and provider-policy checks.

Market risk is handled through paid proof loops. The `$29/week` trial guest pack and `$99/month` Monthly Asset Engine must prove that interview-first capture plus dual-layer extraction creates stronger, more truthful, more routeable assets than ordinary clipping or prompt generation.

Resource risk is handled by reducing concurrency, not amputating the system. If capacity is constrained, CMF may run fewer brands, fewer simultaneous provider jobs, fewer GPU batches, fewer live operators, or narrower validation corpora. It must not remove consent, lineage, Brand Genesis, extraction, routing, receipts, approvals, Publishing Intent, memory, or legacy migration governance.

### Release Gate

No public release is allowed until one Operator can complete a real brand cycle through the documented system without manual database edits, direct provider tinkering, direct legacy runtime imports, approval bypass, unsupported formats, or hidden prompt surgery.

The acceptance loop is:

`intake -> consent -> Brand Genesis -> Interview Intelligence -> Complete Expression Session -> extraction -> archetype routing -> Complete Editing Session -> render -> review -> revision -> approval -> Publishing Intent -> Publer scheduling -> memory update`

The final scope standard is: full documented CMF now; fewer concurrent operations if necessary, but no broken chain.

**Product Requirement Description:** CMF STUDIO must execute governed rendering across deterministic and generative routes without turning provider calls into hidden scripts. ## Non-Functional Requirements

### Performance and Operator Responsiveness

- NFR1: Non-render PWA actions for active brand workspaces must complete within 2 seconds at p95 when no external provider call is required.
- NFR2: Operator dashboards must show current queue, approval, consent, render, and publishing state within 4 seconds at p95 for an active brand workspace.
- NFR3: Telegram approval, rejection, preview, and status commands must return acknowledgement within 3 seconds at p95 when the command is accepted for asynchronous execution.
- NFR4: Long-running jobs such as transcription, provider generation, rendering, evaluation, and publishing must run asynchronously without blocking Operator review workflows.
- NFR5: Render previews must be available in compressed review form before final publishable exports when the renderer supports preview tiers.
- NFR6: Live or post-session review must keep transcript, source playback, timestamp, Expression Moment, route, and evaluation context usable without forcing full media reprocessing.

### Reliability and Workflow Recovery

- NFR7: Every state-changing command must be idempotent and must produce either a successful domain event or a recorded failure.
- NFR8: Complete Expression Session, Complete Editing Session, Brand Genesis, provider job, render, approval, publishing, and memory workflows must survive process restarts without losing completed steps.
- NFR9: Source artifacts, locked Brand Context Versions, approved Expression Moments, provider receipts, evaluation receipts, approval events, and publishing results must be immutable after completion.
- NFR10: Provider failures must support retry, fallback, terminal failure, or manual review states without corrupting lineage.
- NFR11: Batch GPU jobs must checkpoint per asset so partial completion is preserved when a worker stops or fails.
- NFR12: The system must support recovery from provider outage, renderer failure, queue interruption, and object-storage retry without direct database edits.

### Security, Privacy, and Consent Governance

- NFR13: Brand data must be isolated by organization and brand scope across storage, commands, events, provider jobs, receipts, memory, and projections.
- NFR14: Source recordings, transcripts, likeness assets, voice assets, provider tokens, and publishing credentials must be encrypted at rest and in transit.
- NFR15: Asset access must use scoped, short-lived access mechanisms and must not expose private source or render assets publicly by default.
- NFR16: Consent policy must be evaluated before recording use, provider processing, likeness generation, Voice-DNA Boost, memory admission, publishing intent, and Publer scheduling.
- NFR17: High-impact commands must require permission and confirmation according to role, object state, consent, cost, and publishing policy.
- NFR18: Pi, DSPy, Telegram, and UI surfaces must not bypass the Command Bus, consent checks, approval checks, or audit logging.

### Scalability and Resource Governance

- NFR19: The system must scale by brand, queue, worker profile, provider capability, and resource budget rather than by removing required workflow steps.
- NFR20: Capacity constraints must degrade by reducing concurrency, queue priority, render tier, or active brand throughput, not by bypassing consent, lineage, evaluation, approval, or receipts.
- NFR21: GPU workers must operate batch-first: start for queued work, process typed jobs, checkpoint outputs, write receipts, report cost, upload artifacts, and shut down.
- NFR22: Provider and render jobs must carry cost policy, retry policy, timeout policy, and idempotency keys.
- NFR23: The system must expose queue depth, blocked jobs, active workers, failed jobs, estimated cost, and render tier to Operators.
- NFR24: Object storage and derived projections must support rebuild or regeneration from canonical contracts, events, receipts, and immutable artifacts.

### Integration and Provider Interoperability

- NFR25: External integrations must be accessed through capability contracts, not direct domain-level provider SDK calls.
- NFR26: Provider requests and responses must produce receipts containing provider identity, model or workflow version, parameters, source hashes, output hashes, cost, status, and failure details.
- NFR27: Ideogram 4 outputs must preserve `CompositionJob` JSON, prompt hash, composition plate URI, composition analysis, and downstream usage state.
- NFR28: GPT Image 2, Flux 2 Klein 9b, Qwen-Image-Layered, SAM3, self-hosted ComfyUI Docker workers, LavaSR, MOSS-TTS, Remotion, Motion Canvas, Telegram, transcript providers, and Publer must remain replaceable behind contracts.
- NFR29: Publer integration must never become the system of record for approval, caption truth, publishing authority, or asset lineage.
- NFR30: Integration failure must return actionable status to the Operator and must not create duplicate publishing, duplicate provider jobs, or orphaned render outputs.

### Observability, Auditability, and Reproducibility

- NFR31: Every production output must be traceable from source recording and transcript through Interview Asset Contract, Expression Moment, Archetype Route, Asset Package Spec, Complete Editing Session, SceneSpec, provider jobs, render output, evaluation receipt, approval event, and Publishing Intent.
