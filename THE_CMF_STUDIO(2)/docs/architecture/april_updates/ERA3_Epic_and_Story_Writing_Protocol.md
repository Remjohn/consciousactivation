# Era 3 Epic & Story Writing Protocol — Context Generation Pipeline

> [!CAUTION]
> **This document is the MANDATORY protocol for generating Epics and User Stories using the BMAD Agent pipeline. This MUST be executed BEFORE writing Era 3 Tech Specs to ensure all specs are anchored in concrete user actions and have full implementation context.**

---

## 1. The Core Philosophy: Why Stories Before Specs?

Tech Specs can easily drift into abstract, disconnected system design if not grounded in actual user behavior. By enforcing the creation of User Stories *first*, we guarantee that every FastAPI route, every Supabase table modification, and every Experience Primitive orchestration is explicitly tied to a user action that delivers verifiable value. 

The Epics and Stories provide the exact functional and behavioral boundaries that the Tech Specs will architecturally support.

---

## 2. The BMAD Two-Agent Pipeline

We use a strict two-pass pipeline utilizing the existing BMAD agent framework to transform broad PRDs into granular, tech-ready specs.

### Pass 1: Strategic Decomposition (The PM)
*   **Agent:** PM (John) - `bmad/.bmad/bmm/agents/pm.md`
*   **Workflow:** `*create-epics-and-stories`
*   **Goal:** Ingest the PRD, Architecture documents, and Source of Truth. Group Functional Requirements (FRs) into User-Value-driven Epics.
*   **Output:** `epics.md` (The comprehensive breakdown and FR coverage map).

### Pass 2: Tactical Drafting (The SM)
*   **Agent:** SM (Bob) - `bmad/.bmad/bmm/agents/sm.md`
*   **Workflow:** `*create-story` (Executed via `*yolo` mode for rapid drafting without endless elicitation)
*   **Goal:** Take each story from the `epics.md` and create a dev-ready implementation plan. Applies the Exhaustive Validation Checklist to prevent omissions.
*   **Output:** Individual `story_{epic_num}_{story_num}.md` files.

---

## 3. Mandatory Pre-Flight Checklist (For the PM)

Before invoking PM John to run the `*create-epics-and-stories` workflow, ensure the workspace has the following context loaded or explicitly referenced:

1. **The PRD Module:** e.g., `PRD_06_Conscious_Reactions.md`.
2. **The Product Source of Truth:** e.g., `Conscious_Reactions_Source_of_Truth.md`.
3. **The Orchestration Architecture:** e.g., `Conscious_Reactions_Experience_Primitive_Orchestration_Architecture.md`.
4. **Relevant Foundational Specs:** If the stories involve a Mini App, reference the Host Shell Spec (`FR-ERA3-08`).

---

## 4. Epic Design Doctrine (PM Requirements)

> [!IMPORTANT]
> **Epics MUST deliver User Value, not just technical capability. They must not be organized by technical layers.**

*   ✅ **CORRECT (Value-Based):** "Debate Mode Ecosystem", "Audience Jury Participation", "Redemption Round Comeback".
*   ❌ **WRONG (Technical Layer):** "PostgreSQL Schema Update", "FastAPI Routes", "React UI Components".

**Epic Output Requirements:**
1.  **Epic Title:** Value-based and outcome-oriented.
2.  **PRD Coverage:** Explicit mapping to specific Functional Requirements (e.g., "Covers FR-06-01, FR-06-03").
3.  **Technical Context:** High-level description of how this Epic leverages the existing CCP architecture (e.g., "Hooks into `content_machine.py` for extraction").
4.  **Experience Context:** Which specific Primitive Families (TRG, FRC, TRB, FBK, PRG, SOC, SFR, PER) govern the behavior of this Epic.

---

## 5. Story Drafting Doctrine (SM Requirements)

Each individual story generated by SM Bob MUST adhere to the following structure to be useful for Tech Spec generation:

1.  **The Core Format:** `As a [Role], I want [Action], so that [Benefit].`
2.  **Acceptance Criteria (BDD):** `Given` / `When` / `Then` format.
3.  **Technical Notes (Crucial for downstream Tech Specs):**
    *   Specific API endpoints to call or create.
    *   Specific database tables to read/write (e.g., Supabase `sessions`, `vocal_delivery`).
    *   Specific DPA branding tokens to apply.
4.  **Experience Primitive Mapping:**
    *   Identify the exact YAML primitive (e.g., `EXP-FBK-001: RIM Feedback Discipline`) that governs the interaction moment.

---

## 6. Disaster Prevention (The SM Validation Checklist)

During the `*create-story` workflow, SM Bob utilizes a rigorous validation checklist (`checklist.md`) to prevent common LLM hallucination failures. The story MUST pass these checks:

*   **Reinvention Prevention:** Reusing existing CCP pipelines (e.g., `dpa_engine.py`, `learning_path_builder.py`) instead of inventing new arbitrary services.
*   **Architecture Violations:** Ensuring correct API contracts and database isolation (ADR-01 Coach Isolation / RLS).
*   **Primitive Drift:** Ensuring the story doesn't violate the assigned experience primitives (e.g., introducing a 5-step form where a `FRC` friction-reduction primitive demands zero-friction tunneling).

---

## 7. Handover to Tech Spec Writing

Once the Epics and Stories are completed via the BMAD pipeline:

1.  The generated `epics.md` becomes the functional anchor for the upcoming Tech Spec.
2.  The individual stories dictate the exact API routes, UI components, and state transitions that the Tech Spec must architecturally define.
3.  Proceed directly to the `ERA3_Tech_Spec_Writing_Protocol.md` to draft the actual specification.
