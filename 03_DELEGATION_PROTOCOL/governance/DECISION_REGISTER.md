# Decision Register

All 16 product decisions are locked. Architecture may refine mechanisms but may not reverse these product boundaries without a governed PRD amendment.

## D001 — Governed contract protocol with deterministic boundary services

**Decision:** The Delegation Layer is a versioned protocol plus deterministic boundary services. It enforces shared contracts, authority, compatibility, lifecycle, idempotency, routing, audit and result validation without becoming a third creative or production authority.

**Feature:** `F01`

## D002 — Exclusive Content Harness ownership of Visual Asset Demand

**Decision:** The Content Harness exclusively owns and versions demand meaning, Activative function, sequence and asset roles, composition intent, identity, continuity, wrong-reading locks, delivery requirements and authorized budget. Accepted demand versions are immutable; the editor may derive plans or propose amendments only.

**Feature:** `F02`

## D003 — Immutable typed contract family with common envelope

**Decision:** Every interaction is a versioned, single-purpose immutable contract carried by a common delegation envelope with identity, correlation, causation, authority, idempotency, timestamps, references and hashes.

**Feature:** `F03`

## D004 — Stable protocol-owned external lifecycle

**Decision:** The protocol owns a small stable cross-product lifecycle projected from authoritative product messages while each product retains its richer internal state machine.

**Feature:** `F04`

## D005 — Typed supersession and selective invalidation

**Decision:** A superseding demand declares changed authority fields. The protocol validates ownership and prevents stale promotion; the editor returns deterministic impact analysis, preserves valid work and resumes from the smallest affected production point.

**Feature:** `F05`

## D006 — Split production acceptance and consumption acknowledgement

**Decision:** The Visual Asset Editor certifies production validity and demand compliance. The Content Harness or composition runtime separately acknowledges that the result is current and compatible with downstream state without duplicating visual evaluation.

**Feature:** `F06`

## D007 — Split budget authority and execution allocation

**Decision:** The Content Harness authorizes the budget envelope and policy intent. The Visual Asset Editor allocates candidates, compute, evaluators and repairs inside it. The protocol enforces immutable escalations and receipts.

**Feature:** `F07`

## D008 — Authoritative cancellation with safe checkpointing

**Decision:** The Content Harness may cancel delegated work. The editor stops at a safe boundary, prevents stale promotion, preserves useful evidence and returns a disposition and compute receipt. Deadline and race semantics are deterministic.

**Feature:** `F08`

## D009 — Protocol-owned failure taxonomy

**Decision:** Every unsuccessful condition uses a stable protocol failure family and code with responsibility, decision authority, retry semantics, quality-round impact, invalidation scope, terminality and next action.

**Feature:** `F09`

## D010 — Typed Delegation Sets for related assets

**Decision:** Related visual assets remain independent demands and immutable lineages but may be grouped by a Delegation Set defining shared continuity, dependencies, completion policy, set-level evaluation and selective invalidation.

**Feature:** `F10`

## D011 — Governed compatibility negotiation and migration

**Decision:** Products publish compatibility manifests. The protocol negotiates and pins safe versions and features, permits only deterministic lossless adapters, creates immutable migrations and rejects semantic best-effort parsing.

**Feature:** `F11`

## D012 — Typed non-binding amendment proposals

**Decision:** The editor may propose field-level changes with evidence, authority class, predicted effects, reuse and invalidation. Only the owning authority may approve; accepted changes create new immutable demand versions.

**Feature:** `F12`

## D013 — Immutable results with post-completion authorization events

**Decision:** Completed results and assets remain immutable historical facts. Typed invalidation, revocation, supersession, replacement and recall-review events govern current and future consumption while preserving reproducibility and impact analysis.

**Feature:** `F13`

## D014 — Authenticated principals and signed immutable messages

**Decision:** Every state-changing action is attributable to a registered principal and must pass identity, signature, hash, replay, version, schema, authority and lifecycle validation before persistence and routing. An append-only audit chain records the result.

**Feature:** `F14`

## D015 — Control Tower projections, SLOs and conformance suites

**Decision:** Delegation is observable inside the validated Harness Control Tower and protected by protocol SLOs plus executable producer, consumer, authority, lifecycle, compatibility, resilience and Format 02 conformance tests.

**Feature:** `F15`

## D016 — Formal Delegation Implementation Authorization Gate

**Decision:** The PRD authorizes Architecture only. Implementation requires a complete contract family, authority matrix, executable lifecycle, compatibility proof, Format 02 fixtures, integrity controls, resilience tests, Control Tower projection, shared conformance suite and Delegation Development Capsule.

**Feature:** `F16`
