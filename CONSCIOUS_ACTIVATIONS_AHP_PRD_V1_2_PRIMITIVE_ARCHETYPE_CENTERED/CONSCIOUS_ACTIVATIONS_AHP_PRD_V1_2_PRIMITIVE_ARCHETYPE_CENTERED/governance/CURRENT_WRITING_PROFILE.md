# Conscious Activations PRD, Epic, Story, and Tech-Spec Writing Profile

**Profile ID:** `CA-SPEC-WRITING-PROFILE-V1`  
**Status:** `DRAFT_FOR_HUMAN_RATIFICATION`  
**Derived from:** the supplied Specs Builder Library, current Activative Intelligence doctrine, current product boundaries, the CMF Studio predecessor audit, and the Source-First Interview Production amendment.

## Non-negotiable doctrine loading sequence

Before writing or hardening any content-bearing FR, Epic, Story, or Tech Spec, load and reconcile:

1. Interview-First and Expression-First doctrine.
2. Matrix of Edging, including broad signal, candidate survival, Coalition Signature, Edge Product, and Anti-Centroid laws.
3. Meaning and Experience Primitive Families, bindings, misuse modes, suppression conditions, conflicts, and Primitive Coalition Contract.
4. Current archetype and asset-derivative registries.
5. Brand Genesis Session, active Brand Context Version, Guest Voice DNA, and Visual DNA when applicable.
6. RSCS, CCV, CBAR, SDA, SFL, and the required Distillation Layers.
7. Composition-Before-Editing, Visual Research Before Generation, Real-Life Reference Before Generation, Source Fidelity, Negative Space, and Edge Integrity.

A content-bearing requirement is incomplete if it does not explain how the viewer receives a psychological role inside a tension, which primitive coalition builds the recipe, which archetype geometry organizes it, and which Final Script is approved before composition.

**Operator source authority law:** The operator supplies and owns source authorization and intended use. Specifications shall preserve provenance and operator scope but shall not introduce externally imposed creative-filtering gates that flatten the approved creative direction.

## 1. Purpose

This profile preserves the discipline of the supplied PRD, Epic/Story, CBAR, and Tech Spec methods while replacing obsolete CCP/Era-specific constants with the current Conscious Activations authority. It governs this package and should govern later specification factories unless Program Control publishes a superseding profile.

The method is intentionally brownfield. Writers do not begin by imagining an ideal greenfield product. They first establish current authority, read the real contracts and source, inspect accepted tests and failures, identify predecessor value, and state exact ownership boundaries. Product requirements describe user and system behavior. Technical specifications describe implementation. External repositories provide bounded evidence and capability patterns; they do not become product authority.

## 2. Mandatory loading order

Before modifying a governed product requirement or writing a Tech Spec, the writer must load, in order:

1. the current Activative Intelligence Constitution and Program Control authority pointer;
2. the current status and claim ceilings for every affected product;
3. the current product PRD, amendments, decision register, requirement registry, Story inventory, and accepted Tech Specs;
4. current contracts and source code for the behavior being changed;
5. the exact predecessor files, tests, receipts, and migration disposition;
6. the Canonical Interview Source Package and source-first amendment when the behavior is interview-derived;
7. cross-product contracts and ownership boundaries, including Delegation and VAE;
8. external-project evidence only for the bounded capability under study;
9. current blockers, source authority, licenses, evaluation evidence, and implementation gates.

A writer may not infer the content of an unread file from its name or from a previous summary. Missing or unavailable sources must be listed as blockers rather than silently reconstructed.

## 3. PRD module standard

A feature module must be independently readable and must answer ten questions:

1. What product outcome does the feature create?
2. Which user or system job requires it?
3. Which canonical objects and state transitions does it introduce or consume?
4. Which product owns meaning, execution, evaluation, and approval?
5. Which brownfield code and contracts are reused, adapted, replaced, or archived?
6. What workflow and downstream consumers depend on it?
7. What are the Functional Requirements and their behavior-specific acceptance scenarios?
8. What human authority, model, tool, runtime, and external-product boundaries apply?
9. How is the feature verified, evaluated, repaired, replayed, and invalidated?
10. Which risks, stop conditions, and implementation gates remain?

Depth is measured by implementation clarity, not decorative prose. A small feature may be shorter than a broad runtime feature, but no module may substitute generic acceptance language for behavior-specific conditions.

## 4. Functional Requirement card standard

Every Functional Requirement must contain:

- stable ID and outcome-oriented title;
- owning product and feature;
- normative requirement;
- trigger and preconditions;
- required output or state transition;
- explicit invariants and authority limits;
- a positive Given/When/Then scenario;
- an adversarial or denial Given/When/Then scenario;
- required evidence and downstream consumer;
- governing decisions and exact source references;
- lifecycle state: preserved, amended, new, superseded, or deferred.

An FR is invalid when it merely says that a contract exists, invalid inputs fail, and a receipt is emitted. Those are cross-cutting laws. The FR must state what valid behavior actually does, which invalid cases matter, and how a downstream module would fail if the behavior were wrong.

## 5. Value-based Epic standard

Epics are named for user or product value, not internal components. “Build transcript database,” “implement Remotion adapter,” or “create queues” are implementation themes, not value Epics. A valid Epic produces a coherent outcome such as admitting a trusted source, launching a source-backed batch, producing a source-led short, or supervising and correcting a campaign.

Each Epic must name:

- the user/system value;
- the entry and terminal states;
- the major product boundaries;
- the vertical Stories that incrementally deliver useful behavior;
- the evidence needed to claim the Epic outcome.

## 6. Vertical Story standard

Every Story uses:

> As a [specific actor], I want [observable behavior], so that [product value].

A Story owns one vertical outcome and one primary set of FRs. It must not be a technical layer with no usable result. Acceptance criteria use Given/When/Then and include:

- the primary successful journey;
- at least one invalid, stale, unauthorized, or contradictory input;
- one recovery, replay, or selective invalidation case where applicable;
- one CBAR tension case naming the competing goals;
- one downstream failure scenario naming the exact consumer and what breaks if the Story output is wrong.

Every FR has exactly one primary Story owner. Secondary dependencies are permitted but do not duplicate primary ownership.

## 7. CBAR hardening standard

Constraint-Based Adversarial Reasoning is applied after a Story draft. It asks what happens when two individually reasonable goals conflict.

For each Story, the hardening record must identify:

- the protected product law;
- the tempting optimization or shortcut;
- the adversarial situation;
- the incorrect output a weak implementation would produce;
- the exact downstream consumer;
- the concrete downstream failure;
- the gate, test, evidence, or authority that prevents the failure.

Examples in the current product include: pacing versus source truth; semantic similarity versus authority; visual polish versus wrong-reading locks; cost reduction versus silent runtime substitution; batch reuse versus source-version drift; automatic learning versus operator source authorization; operator convenience versus canonical-state integrity.

## 8. Tech Spec standard

A Tech Spec is written only after controlling FRs and Stories are accepted. It contains:

1. exact files and authorities read;
2. problem, solution, scope, and non-goals;
3. architecture traceability and product ownership;
4. existing backend integration with exact paths, schemas, tables, routes, tests, and migration decisions;
5. staged implementation plan;
6. primary schemas, interfaces, and state transitions;
7. backward compatibility, migration, fallback, rollback, and invalidation;
8. implementation tasks with path ownership;
9. behavior-specific acceptance criteria, each including a failure example and Story/CBAR reference;
10. testing, evidence, observability, security, performance, and release strategy.

A Tech Spec may not design a second canonical object when a current contract already owns the behavior. It must explain how existing code is reused or why replacement is necessary.

## 9. Source-first interview laws

Any requirement touching interview-derived content must preserve these laws:

- Activative Interview Brief-led and imported interview paths are both first-class.
- Planned tags and actual expression evidence remain separate.
- The talking-head interview video, transcript, speakers, timestamps, tags, Expression Moments, keyframes, source authority, and provenance form the canonical source package.
- Source-led shorts use the original talking-head footage as A-roll.
- Carousels, SuperVisuals, and animation derivatives preserve exact source and transformation lineage.
- Quotes, condensations, generated voice, and source audio are truthfully labeled.
- Source authority and operator source authorization govern production, publication, retrieval, and learning independently.

## 10. Product-boundary laws

- Builder owns the AtomicHarnessDefinition; Pipeline executes it.
- Interview Expression owns source expression preparation, discovery, validation, and the Asset Package Spec.
- Pipeline owns content-batch orchestration, Workflow Nodes, runtime programs, state, receipts, evaluation control, and selective repair.
- VAE owns visual production methods, providers, SAM3, Lucida, ComfyUI, layered generation, and GNM.
- Delegation owns typed cross-product transport and lifecycle boundaries.
- Studio owns noncanonical projections, typed commands, human review, and publication surfaces.
- Program Control owns cross-product authority, status, release claims, and Format 02 activation.
- Renderers realize accepted programs and never own meaning.

## 11. Model and retrieval laws

A Programmed Model is a versioned artifact plus a bounded learned capability claim, Model Program binding, tools, context, validators, evaluator, fallback, lifecycle, and evidence. Fine-tuning alone is not the product.

Retrieval applies deterministic eligibility before lexical, dense, graph, image/composition, or VLM ranking. Embeddings improve recall and relevance; they never determine authority. Every learned node receives Minimum Complete Context and records a retrieval receipt.

## 12. Completion checklist

A planning package passes only when:

- every feature has a real module;
- every FR has behavior-specific positive and negative acceptance;
- every FR has one primary Story owner;
- every Story has CBAR tension and downstream proof;
- every FR has one primary candidate Tech Spec owner;
- every brownfield source has a disposition;
- cross-product ownership is explicit;
- Format 02 deferral is mechanically enforced;
- source-first release evidence is defined;
- claims remain draft until human ratification;
- no planning artifact is represented as implementation authorization.
