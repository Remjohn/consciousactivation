# F23 — Source-Backed Content Batch and Archetype Routing

## 1. Product claim and user outcome

**User outcome:** One source package launches a coordinated set of eligible derivative Harness jobs with shared analysis and exact lineage.

Compile batch programs, rank supported archetypes, create immutable derivative jobs, reuse source analysis, represent bounded roles, and selectively invalidate. This feature exists to convert doctrine and source evidence into behavior that can be implemented, tested, denied, replayed, and audited. It does not grant implementation, provider, model-training, publication, production, or certification authority by itself.

## 2. Product boundary and ownership

Pipeline owns batch compilation and execution; each derivative Harness remains Builder-defined.

The feature must preserve the current product topology. Builder supplies the immutable Atomic Harness; Pipeline executes its declared nodes; Interview Expression owns source expression preparation and approval; VAE owns visual production; Delegation owns cross-product envelopes; Studio projects state and emits typed operator commands; Program Control owns cross-product authority and release truth. External renderers and models are embodiments, not sources of meaning.

## 3. Canonical objects and state

**Cross-cutting Activative law:** Content activates when it gives the viewer a psychological role inside a tension. Content-bearing work inherits the approved archetype coalition, Primitive Coalition Contract, Final Script, Source Fidelity, Negative Space, Edge Integrity / Anti-Centroid, and Brand/Voice/Visual context when applicable.

The feature consumes current, versioned upstream records and emits typed outputs that remain linked to the active Harness, source package where applicable, Workflow Node, JIT Capsule, execution binding, evaluator, and receipt. Canonical records use stable serialization and immutable identities. A UI projection, model response, provider job, renderer workspace, or local cache cannot become accepted product state until its contract and authority gates pass.

Core invariants for this feature are:

- No route is manufactured.
- Each job has its own Harness and evidence.
- Shared changes invalidate only dependents.

## 5. Workflow integration

Source or Asset Package Spec → target outputs → archetype and Harness eligibility → immutable derivative jobs → explicit role/node graph → coordinated batch execution.

Every handoff is typed. A downstream node begins only after the producer output passes the declared contract and evidence gates. Failures are attributed to source, knowledge, retrieval, Skill, recipe, Programmed Model, tool, runtime, product, evaluator, or human-policy layers before repair or escalation.

## 6. Functional Requirements

### FR-133 — Compile one source-backed content-batch program

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The Pipeline shall compile a Content Batch Orchestration Program from one accepted source package or Asset Package Spec, declaring target derivative jobs, Harnesses, shared ingredients, dependencies, budgets, deadlines, review modes, and source-lineage obligations.

**Trigger and preconditions:** An operator selects a source package, objectives, target outputs, and approved Harnesses.

**Required output or state transition:** A versioned batch program and job graph for short videos, Carousels, SuperVisuals, animation derivatives, or other authorized outputs.

**Authority and invariants:**
- The batch does not change source meaning.
- Each job remains independently cancelable and attributable.
- Shared dependencies are explicit.

**Acceptance scenarios**

- **Primary success:** Given one interview and a request for two shorts plus one Carousel, when compiled, then the jobs share source analysis while preserving separate Harness and evaluation requirements.
- **Adversarial or denial case:** Given an output type with no current Harness, when batch compilation runs, then that job is blocked without blocking unrelated eligible jobs.
- **Evidence required:** Batch schema, dependency graph, budget plan, Harness binding validation, and compilation receipt.

**Governing decisions:** D046  
**Exact source references:** SRC-AM-001, SRC-AM-002, SRC-INT-001


### FR-134 — Route approved expression material through eligible archetypes and Harnesses

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Archetype routing shall rank only current, category-eligible, source-supported routes using approved Expression Moments, tags, audience/context evidence, target platform, and Harness availability; it shall expose alternatives and never manufacture a fit.

**Trigger and preconditions:** An Asset Package Spec and target content objectives are available.

**Required output or state transition:** Ranked route proposals with source support, expected viewer function, required ingredients, exclusions, and selected Harness identity.

**Authority and invariants:**
- Archetype route is not source truth.
- Wrong-category and unsupported routes are excluded before ranking.
- Human creative authority may resolve genuine alternatives.

**Acceptance scenarios**

- **Primary success:** Given an Expression Moment that supports a myth-debunk sequence, when routing runs, then it proposes the matching current short-video Harness and explains evidence.
- **Adversarial or denial case:** Given a strong quote that lacks the evidence needed by a data-story route, when routing runs, then that route is excluded.
- **Evidence required:** Eligibility tests, route ranking benchmark, exclusion reasons, and route-selection receipt.

**Governing decisions:** D045, D046, D052  
**Exact source references:** SRC-INT-003, SRC-AM-002


### FR-135 — Create immutable derivative job contracts

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Each batch job shall bind one Harness version, exact source-package and ingredient references, derivative type, target platform/profile, intended Activative and sequence function, output constraints, runtime capabilities, evaluation, source authority, and review policy.

**Trigger and preconditions:** A batch route is selected.

**Required output or state transition:** An immutable ContentDerivativeJob contract and dependency links.

**Authority and invariants:**
- Jobs cannot read arbitrary source material outside their grant.
- Job intent and source references are separately versioned.
- Execution binding cannot mutate the contract.

**Acceptance scenarios**

- **Primary success:** Given a source-led short job, when admitted, then it contains exact source spans or a bounded selection contract and the talking-head A-roll rule.
- **Adversarial or denial case:** Given a job whose requested platform exceeds source authority scope, when admitted, then it is rejected before execution.
- **Evidence required:** Job schema, authority/source authority filters, source-grant validation, and admission receipt.

**Governing decisions:** D044, D046  
**Exact source references:** SRC-CUR-004, SRC-CUR-007, SRC-AM-002


### FR-136 — Share source analysis while preventing duplicate or conflicting derivatives

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The batch runtime shall reuse accepted source transcripts, phrase packs, shot maps, keyframes, and approved moments, deduplicate overlapping derivative candidates, and detect conflicting source uses or near-duplicate outputs before production spend.

**Trigger and preconditions:** Multiple jobs or candidate routes use the same source package.

**Required output or state transition:** Shared read-only source-analysis bindings, deduplication groups, conflict warnings, and accepted diversity plan.

**Authority and invariants:**
- Shared analysis is immutable.
- Deduplication does not erase distinct archetype purposes.
- Conflicts are explicit.

**Acceptance scenarios**

- **Primary success:** Given two proposed shorts using more than the allowed source overlap for the same hook, when batch planning runs, then the system keeps the stronger or requires a differentiated route.
- **Adversarial or denial case:** Given identical transcript analysis requested by three jobs, when scheduled, then it is reused rather than rerun.
- **Evidence required:** Reuse-cache tests, overlap/dedupe report, diversity evidence, and cost savings receipt.

**Governing decisions:** D046  
**Exact source references:** SRC-EXT-023, SRC-AM-002


### FR-137 — Represent Hunters, Analysts, Composers, and Commanders as bounded workflow roles

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The Pipeline shall model Hunter, Analyst, Composer, and Commander responsibilities as explicit node roles with separate capabilities, tools, inputs, outputs, authority, and failure handling rather than hidden personas or a monolithic multi-agent prompt.

**Trigger and preconditions:** A Harness compiles a source-discovery, evaluation, composition, or gate responsibility.

**Required output or state transition:** Role-labeled Workflow Nodes and typed handoffs mapped to actual execution actors.

**Authority and invariants:**
- Role is independent from model or service identity.
- Hunters do not approve; Composers do not self-evaluate; Commanders authorize only assigned transitions.
- Deterministic implementations may fill roles.

**Acceptance scenarios**

- **Primary success:** Given Expression Moment discovery, when the workflow compiles, then Hunter proposals flow to an Analyst and an authorized approval node before routing.
- **Adversarial or denial case:** Given a single model configured to hunt, compose, approve, and evaluate its own output, when ownership validation runs, then the graph is rejected.
- **Evidence required:** Role/actor taxonomy tests, handoff contracts, authority matrix, and node receipts.

**Governing decisions:** D052  
**Exact source references:** SRC-MIG-001, SRC-INT-001, SRC-CUR-010


### FR-138 — Propagate source or batch changes through selective invalidation

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** When a source package, approved moment, route, source authority state, Harness, or shared ingredient is superseded, the batch runtime shall compute and rerun only affected jobs and descendants while preserving accepted unaffected work.

**Trigger and preconditions:** A shared upstream object changes or is revoked.

**Required output or state transition:** Dependency impact report, invalidation events, preserved checkpoints, rerun plan, and operator notification.

**Authority and invariants:**
- No blanket regeneration without dependency evidence.
- Revoked source authority block new use immediately.
- Historical outputs remain auditable.

**Acceptance scenarios**

- **Primary success:** Given one Expression Moment quote correction, when superseded, then only derivatives that use it are invalidated.
- **Adversarial or denial case:** Given an unrelated keyframe correction, when a short uses no such frame, then its accepted render remains valid.
- **Evidence required:** Dependency-graph tests, invalidation receipts, checkpoint reuse, and historical audit.

**Governing decisions:** D044, D046, D047  
**Exact source references:** SRC-CUR-010, SRC-AM-001, SRC-AM-002


## 7. Execution, model, tool, and human responsibilities

Deterministic services own mechanically decidable facts. Programmed Models or bounded Agent Programs may propose or compile behavior only within their evidence-backed claim and node-local tool grants. Independent evaluators inspect judgment dimensions. Humans own new meaning, identity interpretation, unresolved taste, operator source authority, promotion, and publication authority where assigned. The producing model or runtime cannot accept its own output.

When the feature uses a Hunter, Analyst, Composer, or Commander role, the role is an explicit workflow responsibility. It may be implemented by code, a small model, a larger model, or a human. The role name never hides the actual actor or expand its authority.

## 8. Validation and quality gates

Validation must prove both the intended result and the process that produced it. The minimum feature evidence includes batch graph, route eligibility benchmark, job contracts, dedupe/diversity, role/actor tests, and invalidation. Mechanical tests run before expensive model or visual evaluation. Judgment evaluators are calibrated to the exact category and context. Every denial names the cause, owner, and next admissible action. Every accepted mutation emits an immutable receipt and updates only dependent state.

## 9. Risks, mitigations, and stop conditions

| Risk | Concrete consequence | Mitigation |
|---|---|---|
| Authority drift | A predecessor, external project, UI, model, or renderer is treated as current product authority. | Require exact source disposition, current authority validation, and product-boundary tests. |
| False completion | A synthetic URI, dry-run, unverified command, or model self-report is promoted as a real result. | Require actual files, hashes, independent validation, and artifact-class labels. |
| Over-broad automation | A model or agent gains tools, context, or authority beyond the current node. | Use JIT context, node-local tool grants, sandboxes, and explicit escalation. |
| Hidden mutation | Accepted meaning, source, program, or human preference changes without a versioned record. | Use immutable versions, typed commands, receipts, and selective invalidation. |
| Source truth loss | A derived artifact, route, quote, or cut weakens or changes the participant's actual expression. | Require exact source context, quote/voice labels, independent semantic review, and downstream lineage. |

**Stop conditions**

- The behavior would create a second semantic or canonical state owner.
- A required source, authority, contract, primitive coalition, archetype, Final Script, runtime, evaluator, or human decision is unavailable or unverified.
- The implementation would treat migration evidence or an external project as current authority.
- The change would imply real output, production readiness, or certification beyond available evidence.
