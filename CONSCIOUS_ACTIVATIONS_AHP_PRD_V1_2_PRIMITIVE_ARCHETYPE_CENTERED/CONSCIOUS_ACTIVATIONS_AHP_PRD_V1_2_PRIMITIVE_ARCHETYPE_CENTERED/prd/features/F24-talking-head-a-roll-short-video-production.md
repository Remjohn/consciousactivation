# F24 — Talking-Head A-Roll Short-Video Production

## 1. Product claim and user outcome

**User outcome:** The original interview expression remains the semantic and temporal center of every source-led short.

Select truthful source spans, enforce word/audio boundaries, add supporting visual layers, evaluate actual renders, and repair locally. This feature exists to convert doctrine and source evidence into behavior that can be implemented, tested, denied, replayed, and audited. It does not grant implementation, provider, model-training, publication, production, or certification authority by itself.

## 2. Product boundary and ownership

Pipeline owns the canonical VideoEditProgram and temporal orchestration; renderers execute bounded programs.

The feature must preserve the current product topology. Builder supplies the immutable Atomic Harness; Pipeline executes its declared nodes; Interview Expression owns source expression preparation and approval; VAE owns visual production; Delegation owns cross-product envelopes; Studio projects state and emits typed operator commands; Program Control owns cross-product authority and release truth. External renderers and models are embodiments, not sources of meaning.

## 3. Canonical objects and state

**Cross-cutting Activative law:** Content activates when it gives the viewer a psychological role inside a tension. Content-bearing work inherits the approved archetype coalition, Primitive Coalition Contract, Final Script, Source Fidelity, Negative Space, Edge Integrity / Anti-Centroid, and Brand/Voice/Visual context when applicable.

The feature consumes current, versioned upstream records and emits typed outputs that remain linked to the active Harness, source package where applicable, Workflow Node, JIT Capsule, execution binding, evaluator, and receipt. Canonical records use stable serialization and immutable identities. A UI projection, model response, provider job, renderer workspace, or local cache cannot become accepted product state until its contract and authority gates pass.

Core invariants for this feature are:

- A-roll is mandatory.
- Supporting visuals cannot replace source truth.
- Cut correctness and whole-short meaning are both evaluated.

## 5. Workflow integration

Approved Expression Moment and route → A-roll candidate selection → word-safe EDL → supporting tracks → runtime binding → preview render → rendered evaluation → bounded repair.

Every handoff is typed. A downstream node begins only after the producer output passes the declared contract and evidence gates. Failures are attributed to source, knowledge, retrieval, Skill, recipe, Programmed Model, tool, runtime, product, evaluator, or human-policy layers before repair or escalation.

## 6. Functional Requirements

### FR-139 — Require talking-head A-roll as the base timeline for source-led shorts

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Every interview-derived source-led short-video Harness shall preserve the original talking-head footage and source audio as the base temporal spine; B-roll, graphics, evidence, animation, captions, and reframing may support or temporarily interrupt it only under declared functions.

**Trigger and preconditions:** A source-backed short-video derivative job is admitted.

**Required output or state transition:** A VideoEditProgram whose primary A-roll track is composed from exact source spans and whose supporting tracks declare their relation to A-roll.

**Authority and invariants:**
- Generated visuals cannot replace the source expression as if they were the speaker.
- Source voice remains attributable.
- Any full-frame interruption preserves return and context.

**Acceptance scenarios**

- **Primary success:** Given a 45-second interview short, when the program is validated, then the A-roll track covers the approved spoken spine and all non-A-roll intervals have declared support functions.
- **Adversarial or denial case:** Given a fully generated video using a paraphrased script but labeled as an interview-derived short, when validation runs, then the job fails the A-roll requirement.
- **Evidence required:** A-roll coverage report, source-span map, support-layer manifest, and semantic review.

**Governing decisions:** D043, D044  
**Exact source references:** SRC-AM-002, SRC-EXT-017, SRC-EXT-018, SRC-EXT-019


### FR-140 — Select source spans by expression and archetype function

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** A bounded A-roll selector shall choose eligible source spans using approved Expression Moments, phrase context, speaker, audio events, keyframes, target duration, archetype sequence, and must-preserve/must-exclude constraints.

**Trigger and preconditions:** An admitted short-video job requires a concrete source sequence.

**Required output or state transition:** A candidate portfolio of source-span sequences with runtime, quote, beat, reason, and source context.

**Authority and invariants:**
- No unsupported quote synthesis.
- Selection retains enough context to preserve meaning.
- Required reactions and expression tails remain eligible.

**Acceptance scenarios**

- **Primary success:** Given multiple takes or answer sections, when selection runs, then it ranks the sequence that best satisfies the approved archetype while preserving source truth.
- **Adversarial or denial case:** Given a shorter clip that changes the answer's meaning, when compared, then it cannot win on pacing alone.
- **Evidence required:** Source-context comparison, duration/coverage checks, candidate reasons, and independent selection review.

**Governing decisions:** D043, D045, D052  
**Exact source references:** SRC-INT-001, SRC-EXT-017, SRC-EXT-023


### FR-141 — Enforce word-boundary EDL and audio continuity

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Accepted source cuts shall snap to validated word or permitted silence/audio-event boundaries, use declared padding, preserve speaker handoffs and reactions where required, and apply production-correct audio fades and output-time caption mapping.

**Trigger and preconditions:** An A-roll candidate sequence is compiled into an EDL.

**Required output or state transition:** A mechanically validated EDL and audio/caption boundary plan.

**Authority and invariants:**
- No mid-word cuts.
- Timestamp drift is absorbed within declared padding.
- Every cut works on both audio and video.

**Acceptance scenarios**

- **Primary success:** Given a cut at the end of a spoken phrase, when validated, then it lands on a word boundary, applies permitted padding, and does not create an audible pop.
- **Adversarial or denial case:** Given a cut with a clean transcript boundary but a visible mouth discontinuity, when rendered evaluation runs, then it is not accepted solely by transcript validation.
- **Evidence required:** Word-boundary tests, waveform/boundary checks, caption offset validation, and rendered cut inspection.

**Governing decisions:** D043, D053  
**Exact source references:** SRC-EXT-017


### FR-142 — Compile evidence, B-roll, captions, and motion slots as source support

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The short-video Harness shall compile supporting visual treatments from source evidence, approved assets, keyframes, screenshots, diagrams, captions, and bounded motion-block requests, each with a declared narrative/Activative function, timing, geometry, and lineage.

**Trigger and preconditions:** An accepted A-roll sequence identifies moments that require clarification, proof, emphasis, continuity, or visual relief.

**Required output or state transition:** Supporting-track plan and child asset/motion demands.

**Authority and invariants:**
- Decorative novelty is insufficient justification.
- Proof visuals require evidence lineage.
- Face, captions, and source expression safe regions remain protected.

**Acceptance scenarios**

- **Primary success:** Given a numerical claim, when a data callout appears, then it is timed to the claim, cites the evidence object, and clears the subject/caption regions.
- **Adversarial or denial case:** Given B-roll that contradicts the speaker or obscures a key reaction, when evaluation runs, then it fails.
- **Evidence required:** Support-function validation, geometry checks, evidence references, and composition preview.

**Governing decisions:** D043, D044, D053  
**Exact source references:** SRC-EXT-018, SRC-EXT-019, SRC-EXT-017, SRC-CUR-014


### FR-143 — Evaluate rendered cut boundaries and whole-short coherence

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The evaluator shall inspect the actual rendered output at each cut, overlay transition, caption change, and selected checkpoints for visual jumps, audio pops, subtitle occlusion, wrong frames, grade consistency, source meaning, pacing, and archetype coherence.

**Trigger and preconditions:** An encoded preview exists.

**Required output or state transition:** A boundary-level and whole-short evaluation report with time ranges, failure codes, evidence frames/waveforms, responsible layers, and verdict.

**Authority and invariants:**
- Source inspection cannot substitute for rendered-output inspection.
- Technical and semantic failures remain separate.
- Producer self-description is not evidence.

**Acceptance scenarios**

- **Primary success:** Given a preview with a hidden subtitle at one overlay boundary, when evaluation runs, then the exact time and responsible layer are reported.
- **Adversarial or denial case:** Given a technically perfect short that removes the source reaction and weakens the intended expression, when evaluated, then it fails the semantic/Activative gate.
- **Evidence required:** Rendered timeline views, audio analysis, caption checks, VLM/human calibration, and verdict receipt.

**Governing decisions:** D043, D053  
**Exact source references:** SRC-EXT-017, SRC-CUR-018


### FR-144 — Apply bounded short-video repair and operator revision

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Short-video failures or operator corrections shall compile into the smallest valid edit changes, preserve already-valid source selections and relationships, rerender only affected ranges or downstream outputs where possible, and stop after the configured repair limit or a human-owned boundary.

**Trigger and preconditions:** A rendered evaluation or authorized operator request identifies a temporal defect or preference.

**Required output or state transition:** A ChangeRequestProgram, affected-node/range set, preview, new evaluation, and HumanResolutionEpisode where applicable.

**Authority and invariants:**
- Repair cannot rewrite source speech.
- Unaffected accepted nodes/checkpoints are reused.
- Maximum repair rounds are enforced.

**Acceptance scenarios**

- **Primary success:** Given a caption collision at one scene, when repaired, then only the caption/scene render and dependent final export rerun.
- **Adversarial or denial case:** Given three failed repair passes or an unresolved taste choice, when the limit is reached, then the system escalates rather than looping.
- **Evidence required:** Change diff, invalidation map, reused checkpoint report, rerender hashes, evaluation, and human episode.

**Governing decisions:** D047, D049  
**Exact source references:** SRC-EXT-017, SRC-AM-001, SRC-CUR-019


## 7. Execution, model, tool, and human responsibilities

Deterministic services own mechanically decidable facts. Programmed Models or bounded Agent Programs may propose or compile behavior only within their evidence-backed claim and node-local tool grants. Independent evaluators inspect judgment dimensions. Humans own new meaning, identity interpretation, unresolved taste, operator source authority, promotion, and publication authority where assigned. The producing model or runtime cannot accept its own output.

When the feature uses a Hunter, Analyst, Composer, or Commander role, the role is an explicit workflow responsibility. It may be implemented by code, a small model, a larger model, or a human. The role name never hides the actual actor or expand its authority.

## 8. Validation and quality gates

Validation must prove both the intended result and the process that produced it. The minimum feature evidence includes A-roll coverage, source EDL, word/audio checks, support lineage, rendered boundary evaluation, and bounded repair. Mechanical tests run before expensive model or visual evaluation. Judgment evaluators are calibrated to the exact category and context. Every denial names the cause, owner, and next admissible action. Every accepted mutation emits an immutable receipt and updates only dependent state.

## 9. Risks, mitigations, and stop conditions

| Risk | Concrete consequence | Mitigation |
|---|---|---|
| Authority drift | A predecessor, external project, UI, model, or renderer is treated as current product authority. | Require exact source disposition, current authority validation, and product-boundary tests. |
| False completion | A synthetic URI, dry-run, unverified command, or model self-report is promoted as a real result. | Require actual files, hashes, independent validation, and artifact-class labels. |
| Over-broad automation | A model or agent gains tools, context, or authority beyond the current node. | Use JIT context, node-local tool grants, sandboxes, and explicit escalation. |
| Hidden mutation | Accepted meaning, source, program, or human preference changes without a versioned record. | Use immutable versions, typed commands, receipts, and selective invalidation. |
| Source truth loss | A derived artifact, route, quote, or cut weakens or changes the participant's actual expression. | Require exact source context, quote/voice labels, independent semantic review, and downstream lineage. |

**Stop conditions**

- The behavior would create a second semantic or canonical state owner.
- A required source, authority, contract, primitive coalition, archetype, Final Script, runtime, evaluator, or human decision is unavailable or unverified.
- The implementation would treat migration evidence or an external project as current authority.
- The change would imply real output, production readiness, or certification beyond available evidence.
