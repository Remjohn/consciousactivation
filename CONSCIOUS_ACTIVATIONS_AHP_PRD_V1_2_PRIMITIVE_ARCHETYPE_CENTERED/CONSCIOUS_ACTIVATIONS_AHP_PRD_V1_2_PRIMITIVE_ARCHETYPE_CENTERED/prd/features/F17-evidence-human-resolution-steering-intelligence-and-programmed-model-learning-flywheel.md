# F17 — Evidence, Human Resolution, Steering Intelligence, and Programmed Model Learning Flywheel

## 1. Product claim and user outcome

**User outcome:** Every production and human-resolution episode strengthens explicit knowledge and offline learned capability evidence without live uncontrolled mutation.

Capture exact before/after state, attribute causes, build operator-authorized datasets, promote recipes and models through evidence, and retain rollback. This feature exists to convert doctrine and source evidence into behavior that can be implemented, tested, denied, replayed, and audited. It does not grant implementation, provider, model-training, publication, production, or certification authority by itself.

## 2. Product boundary and ownership

The owning product is the Atomic Harness Pipeline unless an FR explicitly assigns Interview Expression, VAE, Delegation, Studio, Program Control, or an external runtime.

The feature must preserve the current product topology. Builder supplies the immutable Atomic Harness; Pipeline executes its declared nodes; Interview Expression owns source expression preparation and approval; VAE owns visual production; Delegation owns cross-product envelopes; Studio projects state and emits typed operator commands; Program Control owns cross-product authority and release truth. External renderers and models are embodiments, not sources of meaning.

## 3. Canonical objects and state

**Cross-cutting Activative law:** Content activates when it gives the viewer a psychological role inside a tension. Content-bearing work inherits the approved archetype coalition, Primitive Coalition Contract, Final Script, Source Fidelity, Negative Space, Edge Integrity / Anti-Centroid, and Brand/Voice/Visual context when applicable.

The feature consumes current, versioned upstream records and emits typed outputs that remain linked to the active Harness, source package where applicable, Workflow Node, JIT Capsule, execution binding, evaluator, and receipt. Canonical records use stable serialization and immutable identities. A UI projection, model response, provider job, renderer workspace, or local cache cannot become accepted product state until its contract and authority gates pass.

Core invariants for this feature are:

- Capture/indexing is automatic; promotion is not.
- Human preference is scoped.
- Source authority and operator source authorization govern learning.

## 5. Workflow integration

Validated upstream state → feature behavior → typed output and receipt → downstream handoff → evaluation, invalidation, and replay.

Every handoff is typed. A downstream node begins only after the producer output passes the declared contract and evidence gates. Failures are attributed to source, knowledge, retrieval, Skill, recipe, Programmed Model, tool, runtime, product, evaluator, or human-policy layers before repair or escalation.

## 6. Functional Requirements

### FR-097 — Persist complete production and human-resolution episodes

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Each accepted, rejected, repaired, escalated, and human-resolved production episode shall link the source package, Harness, run state, JIT Capsule, candidates, exact tool operations, renders, evaluations, decision authority, result, and downstream outcome.

**Trigger and preconditions:** A node, candidate portfolio, operator action, or campaign reaches a decision or terminal state.

**Required output or state transition:** Immutable ProductionEpisode and, where human authority acted, HumanResolutionEpisode records.

**Authority and invariants:**
- UI actions cannot disappear as anonymous state.
- Human scope and rationale remain explicit.
- No episode mutates live models or canonical doctrine.

**Acceptance scenarios**

- **Primary success:** Given an operator moves a cut and accepts the preview, when the action commits, then the episode records before/after program state, exact tool command, affected nodes, evaluation, and acceptance scope.
- **Adversarial or denial case:** Given a manual change with no attributable authority or before-state identity, when persistence is requested, then the change cannot become accepted state.
- **Evidence required:** Episode hash, before/after references, tool/process receipts, human authority, and result lineage.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-CUR-020, SRC-AM-001, SRC-AM-002


### FR-098 — Attribute learning opportunities to the responsible layer

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** The system shall diagnose whether an episode indicates a source, knowledge, retrieval, context, Skill, Steering Recipe, Programmed Model, tool, runtime, VAE, evaluator, or human-policy opportunity before it enters any learning queue.

**Trigger and preconditions:** A production or human-resolution episode is closed.

**Required output or state transition:** A typed attribution record, candidate programming-material dispositions, and excluded interpretations.

**Authority and invariants:**
- Visible symptoms are not automatically assigned to the model.
- Attribution can remain uncertain or multi-causal.
- Promotion requires later evidence.

**Acceptance scenarios**

- **Primary success:** Given a bad crop caused by a wrong geometry measurement, when diagnosis runs, then the episode is attributed to measurement/runtime rather than used to teach a taste model.
- **Adversarial or denial case:** Given a single operator rejection with no reason, when attribution is ambiguous, then the system stores it as scoped preference evidence but does not promote a universal recipe.
- **Evidence required:** Diagnosis report, responsible-layer confidence, excluded alternatives, and human review where required.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-CUR-020, SRC-AM-001, SRC-EXT-001, SRC-EXT-005


### FR-099 — Compile operator-authorized versioned programming-material corpora

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Supervised demonstrations, preference pairs, hard negatives, repair trajectories, retrieval-obedience cases, and revision-program examples shall be built offline from eligible episodes with exact source authority, operator source authorization, provenance, scope, splits, hashes, and invalidation.

**Trigger and preconditions:** A model or evaluator team requests a dataset build for a bounded capability claim.

**Required output or state transition:** Versioned corpus manifest, training/validation/test partitions, exclusions, source authority/operator source authorization scope, and source episode references.

**Authority and invariants:**
- Real-human source authority govern downstream training.
- Protected test data is isolated.
- No live-production mutation occurs.

**Acceptance scenarios**

- **Primary success:** Given approved video-revision episodes, when a corpus is built, then source authority and identity scope are checked and the test split remains unavailable to the producer.
- **Adversarial or denial case:** Given an episode whose operator source authorization does not permit model training, when dataset selection runs, then it is excluded while remaining available for allowed operational audit if permitted.
- **Evidence required:** Dataset manifest, operator-declared source authority report, split hashes, deduplication, leakage tests, and build receipt.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-CUR-020, SRC-AM-001


### FR-100 — Capture human preferences and corrections with explicit scope

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Every operator approval, rejection, candidate selection, natural-language revision, direct manipulation, tool override, taste explanation, and publication decision shall produce a HumanResolutionEpisode whose applicability scope is explicit and nonuniversal by default.

**Trigger and preconditions:** A human takes an authorized action in any Studio surface or review queue.

**Required output or state transition:** A HumanResolutionEpisode with request, before/after state, operations, reasons, authority, learning scope, and candidate dispositions.

**Authority and invariants:**
- One decision never silently becomes universal doctrine.
- Direct UI manipulation and text requests produce equivalent evidence quality.
- Human identity is never synthesized on replay.

**Acceptance scenarios**

- **Primary success:** Given a Carousel slide reorder, when the operator commits it, then the exact order change and stated reason are recorded with category/profile scope.
- **Adversarial or denial case:** Given an operator drag event without a resolved project/run/authority identity, when commit is attempted, then the UI must request resolution rather than writing anonymous state.
- **Evidence required:** Episode record, authority scope, exact changes, tool program, evaluation, and acceptance.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-AM-001, SRC-AM-002


### FR-101 — Promote Steering Recipes only through repeated evidence

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Repeated successful, scoped interventions may advance a Steering Recipe through proposed, experimental, validated, shadow, limited-production, production, deprecated, and retired states with control comparisons, regression cases, human authority, and rollback.

**Trigger and preconditions:** Episode analysis identifies a recurring condition/intervention pattern.

**Required output or state transition:** A versioned Steering Recipe proposal or lifecycle transition with applicability, preserved properties, evidence, controls, failures, and rollback.

**Authority and invariants:**
- A raw winning prompt is not a recipe.
- A single success cannot change defaults.
- Contradictory and failed evidence remains linked.

**Acceptance scenarios**

- **Primary success:** Given ten comparable hierarchy repairs and controls showing consistent improvement, when promotion is reviewed, then the recipe may enter shadow with exact scope and rollback.
- **Adversarial or denial case:** Given one visually appealing result and no control or failure cases, when promotion is requested, then the recipe remains proposed.
- **Evidence required:** Recipe evidence ledger, control comparison, regression suite, human promotion receipt, and rollback package.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-CUR-020, SRC-AM-001


### FR-102 — Prohibit uncontrolled online mutation while enabling automatic evidence capture

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Production may automatically append immutable episodes, retrieval indexes, and programming-material eligibility records, but it shall not rewrite model weights, canonical Skills, Steering Recipes, authority rules, or production defaults without an offline evaluated release and human authorization.

**Trigger and preconditions:** A production episode or human correction completes.

**Required output or state transition:** Automatic evidence/index updates and, separately, an offline promotion candidate if eligible.

**Authority and invariants:**
- Capture is automatic; promotion is not.
- Historical evidence remains immutable.
- Release rollback is always possible.

**Acceptance scenarios**

- **Primary success:** Given an accepted correction, when the episode closes, then retrieval can find it immediately while the production model remains unchanged.
- **Adversarial or denial case:** Given a service attempting to update an adapter checkpoint directly from a live click stream, when policy enforcement runs, then the mutation is blocked and incident evidence is recorded.
- **Evidence required:** Mutation-denial tests, index update receipt, model artifact immutability checks, and promotion workflow.

**Governing decisions:** D017, D024, D025, D026, D047  
**Exact source references:** SRC-CUR-020, SRC-AM-001


## 7. Execution, model, tool, and human responsibilities

Deterministic services own mechanically decidable facts. Programmed Models or bounded Agent Programs may propose or compile behavior only within their evidence-backed claim and node-local tool grants. Independent evaluators inspect judgment dimensions. Humans own new meaning, identity interpretation, unresolved taste, operator source authority, promotion, and publication authority where assigned. The producing model or runtime cannot accept its own output.

When the feature uses a Hunter, Analyst, Composer, or Commander role, the role is an explicit workflow responsibility. It may be implemented by code, a small model, a larger model, or a human. The role name never hides the actual actor or expand its authority.

## 8. Validation and quality gates

Validation must prove both the intended result and the process that produced it. The minimum feature evidence includes episode completeness, attribution, dataset lineage, retrieval indexing, shadow replay, recipe/model promotion, and mutation-denial. Mechanical tests run before expensive model or visual evaluation. Judgment evaluators are calibrated to the exact category and context. Every denial names the cause, owner, and next admissible action. Every accepted mutation emits an immutable receipt and updates only dependent state.

## 9. Risks, mitigations, and stop conditions

| Risk | Concrete consequence | Mitigation |
|---|---|---|
| Authority drift | A predecessor, external project, UI, model, or renderer is treated as current product authority. | Require exact source disposition, current authority validation, and product-boundary tests. |
| False completion | A synthetic URI, dry-run, unverified command, or model self-report is promoted as a real result. | Require actual files, hashes, independent validation, and artifact-class labels. |
| Over-broad automation | A model or agent gains tools, context, or authority beyond the current node. | Use JIT context, node-local tool grants, sandboxes, and explicit escalation. |
| Hidden mutation | Accepted meaning, source, program, or human preference changes without a versioned record. | Use immutable versions, typed commands, receipts, and selective invalidation. |

**Stop conditions**

- The behavior would create a second semantic or canonical state owner.
- A required source, authority, contract, primitive coalition, archetype, Final Script, runtime, evaluator, or human decision is unavailable or unverified.
- The implementation would treat migration evidence or an external project as current authority.
- The change would imply real output, production readiness, or certification beyond available evidence.
