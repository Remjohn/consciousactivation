# F19 — Conscious Activations Studio Control Tower and Category Workbenches

## 1. Product claim and user outcome

**User outcome:** Operators inspect source-to-batch production, approve genuine human decisions, request corrections, and replay evidence without becoming full-time editors.

Expose one shared supervisory shell with category-native modules, canonical-program projections, autonomy modes, health, evidence, and typed commands. This feature exists to convert doctrine and source evidence into behavior that can be implemented, tested, denied, replayed, and audited. It does not grant implementation, provider, model-training, publication, production, or certification authority by itself.

## 2. Product boundary and ownership

The owning product is the Atomic Harness Pipeline unless an FR explicitly assigns Interview Expression, VAE, Delegation, Studio, Program Control, or an external runtime.

The feature must preserve the current product topology. Builder supplies the immutable Atomic Harness; Pipeline executes its declared nodes; Interview Expression owns source expression preparation and approval; VAE owns visual production; Delegation owns cross-product envelopes; Studio projects state and emits typed operator commands; Program Control owns cross-product authority and release truth. External renderers and models are embodiments, not sources of meaning.

## 3. Canonical objects and state

**Cross-cutting Activative law:** Content activates when it gives the viewer a psychological role inside a tension. Content-bearing work inherits the approved archetype coalition, Primitive Coalition Contract, Final Script, Source Fidelity, Negative Space, Edge Integrity / Anti-Centroid, and Brand/Voice/Visual context when applicable.

The feature consumes current, versioned upstream records and emits typed outputs that remain linked to the active Harness, source package where applicable, Workflow Node, JIT Capsule, execution binding, evaluator, and receipt. Canonical records use stable serialization and immutable identities. A UI projection, model response, provider job, renderer workspace, or local cache cannot become accepted product state until its contract and authority gates pass.

Core invariants for this feature are:

- Studio read models are nonauthoritative.
- Direct manipulation uses typed command paths.
- Only assigned human authority receives actionable gates.

## 5. Workflow integration

Validated upstream state → feature behavior → typed output and receipt → downstream handoff → evaluation, invalidation, and replay.

Every handoff is typed. A downstream node begins only after the producer output passes the declared contract and evidence gates. Failures are attributed to source, knowledge, retrieval, Skill, recipe, Programmed Model, tool, runtime, product, evaluator, or human-policy layers before repair or escalation.

## 6. Functional Requirements

### FR-109 — Expose source-to-batch run and graph projections

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** The Studio Control Tower shall show the Canonical Interview Source Package, content-batch jobs, Harnesses, phases, Workflow Nodes, bounded roles, owners, status, blockers, costs, artifacts, source-time lineage, and exact execution evidence.

**Trigger and preconditions:** An operator opens a project, source package, campaign, or run.

**Required output or state transition:** Live read models for source, batch, run graph, artifacts, evidence, human gates, and next admissible actions.

**Authority and invariants:**
- Read models do not become authority stores.
- Synthetic/dry-run/real artifact classes are unmistakable.
- Role labels do not hide actual execution actors.

**Acceptance scenarios**

- **Primary success:** Given one interview producing a short, Carousel, and SuperVisual, when the batch board opens, then the operator can trace every output to the same source package and exact source spans.
- **Adversarial or denial case:** Given a UI fixture or stale projection that cannot prove its source state, when displayed, then it is labeled nonauthoritative and cannot drive a command.
- **Evidence required:** Read-model consistency tests, realtime event mapping, source-lineage drill-down, and projection freshness receipts.

**Governing decisions:** D005, D024, D031, D038, D046, D048, D052  
**Exact source references:** SRC-AM-001, SRC-AM-002, SRC-INT-001, SRC-MIG-001


### FR-110 — Provide category-native supervisory previews over canonical programs

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Static, video, interview, VAE, and knowledge/model Studio modules shall display the current canonical programs, source evidence, assets, candidates, evaluations, repairs, and allowed human actions without owning hidden production state.

**Trigger and preconditions:** A run reaches a reviewable state or an operator enters a category-native Studio.

**Required output or state transition:** Interactive projections of Composition IR, VideoEditProgram, source transcript/time map, VAE demands/results, and model/retrieval evidence.

**Authority and invariants:**
- UI state round-trips into typed commands.
- Reopening reconstructs from canonical records.
- Direct manipulation cannot bypass validation.

**Acceptance scenarios**

- **Primary success:** Given a video cut moved by drag, when committed, then the Studio emits the same typed ChangeRequestProgram as an equivalent natural-language request.
- **Adversarial or denial case:** Given a timeline edit stored only in browser memory, when the page reloads, then it is not accepted and the canonical timeline remains unchanged.
- **Evidence required:** Round-trip tests, command receipts, reload/replay tests, and UI/canonical-state consistency.

**Governing decisions:** D005, D024, D031, D048, D049  
**Exact source references:** SRC-AM-001, SRC-MIG-001, SRC-EXT-021, SRC-EXT-022


### FR-111 — Apply Harness autonomy modes and explicit human gates

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Each campaign shall operate in AUTOPILOT, REVIEW_BEFORE_SHIP, CHECKPOINTED, or SHADOW mode, pausing only at declared human authority nodes, genuine exceptions, budget or policy changes, or unresolved taste boundaries.

**Trigger and preconditions:** A campaign is launched or a node encounters a human-owned condition.

**Required output or state transition:** An autonomy-policy binding, human-gate records, exception package, and resume decision.

**Authority and invariants:**
- Mature Harnesses target exception-only supervision.
- Humans are not asked to approve deterministic facts.
- No publication occurs beyond the mode’s authority.

**Acceptance scenarios**

- **Primary success:** Given REVIEW_BEFORE_SHIP mode, when all production nodes pass, then one final acceptance is requested without intermediate pauses unless a genuine exception occurred.
- **Adversarial or denial case:** Given SHADOW mode, when all outputs pass evaluation, then publication remains prohibited.
- **Evidence required:** Autonomy binding, pause/resume logs, authority tests, and publication-denial receipt.

**Governing decisions:** D005, D024, D048  
**Exact source references:** SRC-AM-001, SRC-AM-002


### FR-112 — Inspect retrieval, Skills, recipes, Programmed Models, and exclusions

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** Operators shall be able to inspect the JIT Capsule, retrieved Skills and Steering Recipes, authority filters, exclusions, contradictory evidence, model claim envelope, benchmark evidence, limitations, fallbacks, and the reason each item influenced the current decision.

**Trigger and preconditions:** A learned, retrieval-backed, or taste-bearing node is reviewed.

**Required output or state transition:** A human-readable evidence view linked to exact retrieval and model receipts.

**Authority and invariants:**
- Similarity never appears as authority.
- Protected evaluator internals remain hidden.
- Unavailable context is disclosed.

**Acceptance scenarios**

- **Primary success:** Given a static strategy proposal, when inspected, then the operator sees the selected Skill, two eligible recipes, one relevant negative precedent, and the exact exclusions.
- **Adversarial or denial case:** Given a semantically similar but wrong-category recipe, when the operator inspects retrieval, then it appears only in the exclusion evidence and not the active JIT Capsule.
- **Evidence required:** Retrieval receipt, claim manifest, benchmark links, and UI evidence tests.

**Governing decisions:** D005, D024, D048  
**Exact source references:** SRC-CUR-020, SRC-EXT-027, SRC-EXT-028, SRC-AM-002


### FR-113 — Expose runtime, provider, worker, and budget health

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** The Studio shall expose preflight, compatibility, health, queue, lease, resource, cost, and failure evidence for Skia, PRETEXT, FFmpeg, Remotion, HyperFrames, VAE providers, GNM, workers, and other bound embodiments without treating their presence as production certification.

**Trigger and preconditions:** An operator inspects a binding, campaign, active worker, blocker, or incident.

**Required output or state transition:** Capability and worker health projections, budget usage, exact blocker cause, owner, and next admissible action.

**Authority and invariants:**
- Health does not imply quality certification.
- Credentials and secrets are never exposed.
- Unavailable capabilities cannot be silently emulated.

**Acceptance scenarios**

- **Primary success:** Given a failed Remotion worker preflight, when inspected, then the Studio shows the exact dependency/version failure and blocks only the affected nodes.
- **Adversarial or denial case:** Given a fake provider result or synthetic URI, when health/result views load, then the artifact is clearly nonproduction and cannot satisfy completion.
- **Evidence required:** Health receipts, worker heartbeats, budget ledger, blocker records, and security tests.

**Governing decisions:** D005, D024, D038, D050  
**Exact source references:** SRC-AM-002, SRC-MIG-001, SRC-EXT-018, SRC-EXT-019, SRC-EXT-026


### FR-114 — Export source-to-ship audit and replay packages

**Lifecycle:** `AMENDED_V1_1`  
**Normative requirement:** A completed campaign shall export the source package manifest, content-batch plan, Harnesses and bindings, JIT Capsules, typed programs, HumanResolutionEpisodes, artifacts, evaluations, execution fingerprints, logs, publication evidence, and replay instructions needed to reconstruct the accepted result.

**Trigger and preconditions:** A campaign reaches accepted, rejected, canceled, shadow-complete, or published terminal state.

**Required output or state transition:** A portable audit/replay bundle with exact hashes and claim boundaries.

**Authority and invariants:**
- No secrets or absolute machine paths.
- Historical human decisions are replayed as recorded rather than regenerated.
- External missing dependencies are declared.

**Acceptance scenarios**

- **Primary success:** Given a published short and Carousel, when the audit package is opened elsewhere, then the source lineage and all accepted decisions can be verified even if rendering dependencies are unavailable.
- **Adversarial or denial case:** Given an artifact with no source or execution fingerprint, when export runs, then the bundle fails completeness validation.
- **Evidence required:** Portable bundle validation, hash manifest, secret/path scan, and clean-room replay report.

**Governing decisions:** D005, D024, D038, D044, D047  
**Exact source references:** SRC-AM-001, SRC-AM-002, SRC-MIG-001


## 7. Execution, model, tool, and human responsibilities

Deterministic services own mechanically decidable facts. Programmed Models or bounded Agent Programs may propose or compile behavior only within their evidence-backed claim and node-local tool grants. Independent evaluators inspect judgment dimensions. Humans own new meaning, identity interpretation, unresolved taste, operator source authority, promotion, and publication authority where assigned. The producing model or runtime cannot accept its own output.

When the feature uses a Hunter, Analyst, Composer, or Commander role, the role is an explicit workflow responsibility. It may be implemented by code, a small model, a larger model, or a human. The role name never hides the actual actor or expand its authority.

## 8. Validation and quality gates

Validation must prove both the intended result and the process that produced it. The minimum feature evidence includes read-model consistency, UI round-trip, autonomy/gate tests, role permissions, health projections, and audit exports. Mechanical tests run before expensive model or visual evaluation. Judgment evaluators are calibrated to the exact category and context. Every denial names the cause, owner, and next admissible action. Every accepted mutation emits an immutable receipt and updates only dependent state.

## 9. Risks, mitigations, and stop conditions

| Risk | Concrete consequence | Mitigation |
|---|---|---|
| Authority drift | A predecessor, external project, UI, model, or renderer is treated as current product authority. | Require exact source disposition, current authority validation, and product-boundary tests. |
| False completion | A synthetic URI, dry-run, unverified command, or model self-report is promoted as a real result. | Require actual files, hashes, independent validation, and artifact-class labels. |
| Over-broad automation | A model or agent gains tools, context, or authority beyond the current node. | Use JIT context, node-local tool grants, sandboxes, and explicit escalation. |
| Hidden mutation | Accepted meaning, source, program, or human preference changes without a versioned record. | Use immutable versions, typed commands, receipts, and selective invalidation. |

**Stop conditions**

- The behavior would create a second semantic or canonical state owner.
- A required source, authority, contract, primitive coalition, archetype, Final Script, runtime, evaluator, or human decision is unavailable or unverified.
- The implementation would treat migration evidence or an external project as current authority.
- The change would imply real output, production readiness, or certification beyond available evidence.
