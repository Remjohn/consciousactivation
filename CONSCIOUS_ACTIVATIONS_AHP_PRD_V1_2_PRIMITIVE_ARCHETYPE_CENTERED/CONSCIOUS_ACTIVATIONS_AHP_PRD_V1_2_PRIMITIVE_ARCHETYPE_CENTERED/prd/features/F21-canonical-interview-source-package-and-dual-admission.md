# F21 — Canonical Interview Source Package and Dual Admission

## 1. Product claim and user outcome

**User outcome:** Activative Interview sessions and imported interviews become equally valid, truthfully differentiated roots for downstream production.

Normalize video, transcript, speakers, times, source authority, and provenance into one immutable source object without inventing missing Brief history. This feature exists to convert doctrine and source evidence into behavior that can be implemented, tested, denied, replayed, and audited. It does not grant implementation, provider, model-training, publication, production, or certification authority by itself.

## 2. Product boundary and ownership

Interview Expression owns source admission and package truth; Pipeline consumes the accepted package.

The feature must preserve the current product topology. Builder supplies the immutable Atomic Harness; Pipeline executes its declared nodes; Interview Expression owns source expression preparation and approval; VAE owns visual production; Delegation owns cross-product envelopes; Studio projects state and emits typed operator commands; Program Control owns cross-product authority and release truth. External renderers and models are embodiments, not sources of meaning.

## 3. Canonical objects and state

**Cross-cutting Activative law:** Content activates when it gives the viewer a psychological role inside a tension. Content-bearing work inherits the approved archetype coalition, Primitive Coalition Contract, Final Script, Source Fidelity, Negative Space, Edge Integrity / Anti-Centroid, and Brand/Voice/Visual context when applicable.

The feature consumes current, versioned upstream records and emits typed outputs that remain linked to the active Harness, source package where applicable, Workflow Node, JIT Capsule, execution binding, evaluator, and receipt. Canonical records use stable serialization and immutable identities. A UI projection, model response, provider job, renderer workspace, or local cache cannot become accepted product state until its contract and authority gates pass.

Core invariants for this feature are:

- Planned and actual expression are separate.
- Imported-source absence is explicit.
- Original media and history are immutable.

## 5. Workflow integration

Brief-led or imported source → technical and source authority admission → transcript/speaker/time alignment → immutable Canonical Interview Source Package → downstream expression discovery.

Every handoff is typed. A downstream node begins only after the producer output passes the declared contract and evidence gates. Failures are attributed to source, knowledge, retrieval, Skill, recipe, Programmed Model, tool, runtime, product, evaluator, or human-policy layers before repair or escalation.

## 6. Functional Requirements

### FR-121 — Establish the Canonical Interview Source Package as the batch root

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The product shall represent each interview-derived campaign with one immutable Canonical Interview Source Package that binds the accepted source media, transcript, speakers, timestamps, tags, Expression Moments, keyframes, source authority, and provenance used by every derivative job.

**Trigger and preconditions:** An Activative Interview session completes or an existing interview is uploaded for production.

**Required output or state transition:** A stable source_package_id, version, manifest, source assets, transcript structures, evidence objects, source authority scope, and downstream lineage root.

**Authority and invariants:**
- Every derivative references this root.
- The package records facts and evidence but does not itself create a content strategy.
- Original source bytes and historical versions remain immutable.

**Acceptance scenarios**

- **Primary success:** Given an accepted interview and transcript, when the package is finalized, then a short-video, Carousel, and SuperVisual job can independently resolve the exact source objects they use.
- **Adversarial or denial case:** Given a derivative request with no accepted source_package_id, when batch admission runs, then the request is rejected as ungrounded.
- **Evidence required:** Package schema validation, source/hash manifest, lineage graph, and acceptance receipt.

**Governing decisions:** D041, D042, D044  
**Exact source references:** SRC-INT-001, SRC-INT-002, SRC-INT-003, SRC-AM-002


### FR-122 — Support Activative Interview Brief-led admission

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The Interview Expression integration shall admit sessions produced from an Activative Interview Brief, preserving planned Context Premises, Matrix of Edging, anchors, Interview Asset Contracts, intended tags, operator source authorization, and session design separately from what the participant actually expressed.

**Trigger and preconditions:** An approved Activative Interview Brief and recording session are supplied.

**Required output or state transition:** A source package containing the Brief lineage, planned activation fields, recording assets, observed expression evidence, and planned-versus-observed comparison.

**Authority and invariants:**
- Planned intent is never represented as actual expression.
- Human reaction remains source evidence.
- Unsupported Brief expectations may remain unmet without fabrication.

**Acceptance scenarios**

- **Primary success:** Given a Brief seeking a specific tension, when the interview does not produce it, then the package records the planned tag as unconfirmed and does not create an Expression Moment.
- **Adversarial or denial case:** Given a planned anchor with no matching source span, when downstream routing runs, then it cannot be used as quote or evidence.
- **Evidence required:** Brief hash, session operator source authorization, planned/observed tag comparison, and Interview Expression acceptance receipt.

**Governing decisions:** D041, D042, D045  
**Exact source references:** SRC-INT-001, SRC-INT-002, SRC-INT-003


### FR-123 — Support imported interview and transcript admission as a first-class path

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The product shall admit an existing interview video with an uploaded or generated transcript without pretending that an Activative Interview Brief, Matrix of Edging, planned anchors, or predeclared asset contracts existed.

**Trigger and preconditions:** An operator uploads an interview or talking-head video and transcript, or requests transcription.

**Required output or state transition:** A source package with imported-source provenance, transcript/speaker alignment, retrospective evidence candidates, operator-confirmed context, and explicit absent-planning fields.

**Authority and invariants:**
- Imported sessions receive equal downstream production capability after validation.
- Missing upstream planning is disclosed, not synthesized.
- Retrospective inferences remain labeled until confirmed.

**Acceptance scenarios**

- **Primary success:** Given a legacy interview with no Brief, when admitted, then the system can still produce a trusted source package and launch the same derivative Harnesses.
- **Adversarial or denial case:** Given a model-inferred Matrix of Edging represented as if it had guided the original interview, when validation runs, then the claim is rejected.
- **Evidence required:** Import receipt, source/transcript hashes, absent-field declaration, retrospective-tag evidence, and operator confirmation.

**Governing decisions:** D041, D042, D045  
**Exact source references:** SRC-INT-001, SRC-AM-002, SRC-EXT-017, SRC-EXT-023


### FR-124 — Align transcript, speakers, time bases, and media technically

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The source-package compiler shall reconcile word-level timestamps, phrase boundaries, speaker diarization, audio events, source frame/time bases, media duration, and transcript confidence before source spans become eligible.

**Trigger and preconditions:** Source media and a transcript or transcription result are available.

**Required output or state transition:** Word and phrase records, speaker map, audio-event track, alignment confidence, source-time conversion functions, and blockers.

**Authority and invariants:**
- No source span exceeds media bounds.
- Speaker changes and phrase breaks are reproducible.
- Low-confidence regions remain visible.

**Acceptance scenarios**

- **Primary success:** Given multi-speaker interview audio, when alignment completes, then every eligible phrase references one speaker and exact source times within tolerance.
- **Adversarial or denial case:** Given transcript drift beyond the declared tolerance, when Expression Moment extraction starts, then the affected ranges are blocked or routed for correction.
- **Evidence required:** Alignment benchmark, duration/time-base validation, diarization review samples, and phrase-pack hash.

**Governing decisions:** D041, D044  
**Exact source references:** SRC-EXT-017, SRC-EXT-020, SRC-EXT-023, SRC-INT-001


### FR-125 — Govern operator-declared source authority, operator-defined source handling, and derivative usage

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** The source package shall declare participant operator source authorization, ownership, allowed derivative formats, publication scope, model-training eligibility, identity/voice use, retention, source revocation, and restricted evidence before any job or learning disposition is admitted.

**Trigger and preconditions:** Source material enters the workspace or its permitted use changes.

**Required output or state transition:** A versioned source authority-and-operator source authorization envelope linked to source assets, derivatives, training eligibility, and source revocation/invalidation policy.

**Authority and invariants:**
- No job exceeds the granted scope.
- Training source authority are independent from production source authority.
- Source revocation preserves audit history while preventing new prohibited use.

**Acceptance scenarios**

- **Primary success:** Given operator source authorization for social derivatives but not model training, when the content batch runs, then production may proceed while dataset compilation excludes the source.
- **Adversarial or denial case:** Given expired or withdrawn publication authority, when a publish command runs, then it is denied and affected unpublished derivatives are blocked.
- **Evidence required:** Operator source-authority declaration, provenance checks, scope tests, usage audit, and source-revocation simulation.

**Governing decisions:** D041, D044, D047  
**Exact source references:** SRC-INT-001, SRC-AM-001


### FR-126 — Version, supersede, and selectively invalidate source packages

**Lifecycle:** `NEW_V1_1`  
**Normative requirement:** Corrections to transcripts, speakers, tags, source authority, Expression Moments, or source assets shall create a new immutable source-package version and compute selective downstream invalidation rather than rewriting accepted history.

**Trigger and preconditions:** An operator or validator corrects an accepted source-package field or the operator source-authority scope changes.

**Required output or state transition:** A superseding package version, field-level change set, affected-job graph, invalidation proposals, and historical reproduction path.

**Authority and invariants:**
- Original package versions remain reproducible.
- Only dependent derivatives are invalidated.
- Human decisions are never regenerated during replay.

**Acceptance scenarios**

- **Primary success:** Given a corrected speaker assignment in one phrase, when supersession occurs, then only derivatives using that phrase are invalidated.
- **Adversarial or denial case:** Given a corrected transcript, when a service mutates the original package in place, then integrity validation fails.
- **Evidence required:** Version/hash comparison, dependency analysis, selective invalidation tests, and historical replay.

**Governing decisions:** D041, D044, D047  
**Exact source references:** SRC-CUR-010, SRC-AM-001, SRC-AM-002


## 7. Execution, model, tool, and human responsibilities

Deterministic services own mechanically decidable facts. Programmed Models or bounded Agent Programs may propose or compile behavior only within their evidence-backed claim and node-local tool grants. Independent evaluators inspect judgment dimensions. Humans own new meaning, identity interpretation, unresolved taste, operator source authority, promotion, and publication authority where assigned. The producing model or runtime cannot accept its own output.

When the feature uses a Hunter, Analyst, Composer, or Commander role, the role is an explicit workflow responsibility. It may be implemented by code, a small model, a larger model, or a human. The role name never hides the actual actor or expand its authority.

## 8. Validation and quality gates

Validation must prove both the intended result and the process that produced it. The minimum feature evidence includes dual-admission tests, source/transcript alignment, source authority, versioning, supersession, and source-package receipts. Mechanical tests run before expensive model or visual evaluation. Judgment evaluators are calibrated to the exact category and context. Every denial names the cause, owner, and next admissible action. Every accepted mutation emits an immutable receipt and updates only dependent state.

## 9. Risks, mitigations, and stop conditions

| Risk | Concrete consequence | Mitigation |
|---|---|---|
| Authority drift | A predecessor, external project, UI, model, or renderer is treated as current product authority. | Require exact source disposition, current authority validation, and product-boundary tests. |
| False completion | A synthetic URI, dry-run, unverified command, or model self-report is promoted as a real result. | Require actual files, hashes, independent validation, and artifact-class labels. |
| Over-broad automation | A model or agent gains tools, context, or authority beyond the current node. | Use JIT context, node-local tool grants, sandboxes, and explicit escalation. |
| Hidden mutation | Accepted meaning, source, program, or human preference changes without a versioned record. | Use immutable versions, typed commands, receipts, and selective invalidation. |
| Source truth loss | A derived artifact, route, quote, or cut weakens or changes the participant's actual expression. | Require exact source context, quote/voice labels, independent semantic review, and downstream lineage. |

**Stop conditions**

- The behavior would create a second semantic or canonical state owner.
- A required source, authority, contract, primitive coalition, archetype, Final Script, runtime, evaluator, or human decision is unavailable or unverified.
- The implementation would treat migration evidence or an external project as current authority.
- The change would imply real output, production readiness, or certification beyond available evidence.
